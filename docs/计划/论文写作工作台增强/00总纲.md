# 论文写作工作台增强总纲

## 1. 需求理解

当前 `/writing` 页面已经具备论文草稿编辑器的基本形态：

- 左侧：项目选择、草稿列表、章节结构、完成进度；
- 中间：当前章节编辑区、保存、复制、续写；
- 右侧：AI 建议、引用、交付状态；
- 后端：已有草稿、章节生成、摘要生成、合规检查、答辩大纲等能力。

但它距离 PRD 中“基于真实成果、真实文献、真实实验结果进行论文写作”的要求还有差距。增强重点不应是继续堆 UI，而是让写作时能清楚知道：

- 当前章节依赖哪些文献和项目成果；
- 哪些内容可以写成真实结果，哪些只能写成实验设计；
- 合规检查发现了什么风险；
- 写作完成后如何导出和进入答辩材料。

## 2. 关键假设

- 本轮不新增依赖，不改数据库结构。
- 优先复用已有 API：`listOutcomes`、`listProjectDesigns`、`listDrafts`、`generateChapter`、`checkCompliance`、`getDefenseOutline`。
- 先做前端工作台增强，后端只在发现阻塞时做小修。
- 不引入富文本编辑器，继续使用当前 textarea，避免扩大风险。
- 论文结果章节必须强化“无真实数据时不生成结果”的提示和检查。

## 3. 总体目标

增强后，论文写作页应达到：

- 写作者能在编辑当前章节时看到“本章证据状态”；
- 右侧引用面板能区分项目设计、文献引用、成果材料；
- 合规检查结果不仅显示分数，还显示具体问题、严重级别和建议处理动作；
- 对第 4/5 章这类结果章节，明确提示是否具备真实成果支撑；
- 导出区支持 DOCX/PDF 选择，并显示答辩 PPT 大纲状态；
- 页面不再出现“按钮占位但不可用”的操作，如视觉占位的改写/格式按钮。

## 4. 分阶段实施

### 阶段 1：证据状态与章节写作约束

先增强当前章节上下文展示，不改生成逻辑。

目标：

- 顶部或右侧显示当前章节证据状态；
- 标出本章是否有项目成果、文献引用、项目设计支撑；
- 第 4/5 章无成果时给出“只能写实验设计，不能写实验结果”的明显提示；
- 移除或禁用无实际功能的“改写、B、I、段落”等占位按钮。

### 阶段 2：右侧面板增强

重组右侧 `建议 / 引用 / 交付` 三个标签页。

目标：

- `建议`：基于当前章节类型给出更具体的写作约束；
- `引用`：分组展示项目设计引用、草稿引用、成果材料；
- `交付`：显示导出、合规、答辩 PPT 大纲状态；
- 每个空状态都说明下一步去哪里补资料。

### 阶段 3：合规检查结果可操作化

增强 `checkCompliance` 的前端展示。

目标：

- 显示 `passed`、`overall_score`、issues 列表；
- issue 按 error/warning/info 区分；
- 显示问题类型、位置、描述、证据；
- 提供“复制问题说明”“跳转当前章节编辑”这类轻操作；
- 不在本阶段做用户确认持久化，避免后端扩展。

### 阶段 4：导出与答辩衔接

完善写作完成后的交付入口。

目标：

- 导出按钮支持 DOCX/PDF 格式选择；
- 显示当前草稿章节完成度、成果材料数量、答辩 PPT 大纲状态；
- 若答辩大纲提示真实数据不足，明确展示缺口；
- 保留进入项目详情或项目成果管理的入口。

## 5. 建议修改文件

- `frontend/src/app/writing/page.tsx`
- `frontend/src/lib/api.ts`（如需补导出 URL 参数，不新增依赖）
- `frontend/src/lib/types.ts`（如合规结果类型不完整再补）

## 6. 验收标准

- `/writing` 首屏仍是论文编辑器，不退回流程卡片。
- 当前章节能看到证据状态和真实数据风险提示。
- 无项目成果时，第 4/5 章不会诱导用户生成“实验结果”。
- 合规检查结果能展示具体问题，而不是只显示一个分数。
- DOCX/PDF 导出入口清晰。
- `npm.cmd run build` 通过。
