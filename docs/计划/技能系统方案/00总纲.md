# 技能系统方案总纲

## 1. 需求理解

当前项目已经具备多种专门能力：

- 论文写作 Agent
- 开题 PPT Agent
- 答辩 PPT Agent
- 需求理解 / 文献检索 / 文献分析 / 研究方向 / 项目设计 Agent
- 轻量多 Agent orchestration / workflow 基座

但这些能力目前仍然主要表现为：

1. 直接在 API 中调用具体 Agent；
2. Prompt、规则、输入约束、输出约束和业务流程耦合在一起；
3. 用户前端动作与后端 Agent 的映射关系偏硬编码；
4. 论文写作与 PPT 生成虽然已经是“专门能力”，但还不是“可注册、可路由、可替换、可约束”的技能系统。

本方案的目标不是把项目做成 Codex 插件格式本身，而是在当前项目中实现一层 **类 skill 的能力注册与调度系统**，优先服务：

1. 论文写作
2. PPT 制作

## 2. 关键判断

### 2.1 当前项目已经有“能力模块”，但还没有“技能系统”

当前已有：

- `paper_writing_agent.py`
- `ppt_agent.py`
- `defense_ppt_agent.py`
- `proposal_agent.py`
- `project_design_agent.py`

说明：

这些是业务能力模块，不是 skill registry。

### 2.2 当前项目已经有 workflow 基座，可直接复用

当前已有：

- `backend/app/agents/orchestration/node.py`
- `backend/app/agents/orchestration/state.py`
- `backend/app/agents/orchestration/runner.py`

说明：

新的技能系统不应绕开这套基座重做，而应建立在它之上。

### 2.3 当前阶段不适合直接照搬 Codex skill 格式

原因：

1. Codex skill 本质上是“给代理看的操作说明”
2. 你的项目需要的是“产品内可运行的技能定义与执行层”
3. 直接照搬 `.codex` skill 目录格式，对项目运行时价值不大

## 3. 目标

新增一层技能系统，使项目达到以下状态：

1. 论文写作与 PPT 制作能力可以按 `skill_id` 被调用
2. 每个 skill 都有结构化定义，不再只有散落 prompt
3. 每个 skill 都带输入约束、输出约束和 guard 规则
4. skill 可以被 router 选择，也可以被前端显式指定
5. skill 执行过程仍通过当前 workflow / orchestration 层运行

## 4. 推荐方案

推荐采用：

**轻量 Skill Registry + Skill Definition + Skill Executor + Skill Router + Skill Guards**

并明确建立在当前 `orchestration` 基座上。

## 5. 技能系统分层

### 5.1 Skill Definition

定义一个技能“是什么”。

每个 skill 至少包含：

- `id`
- `name`
- `description`
- `domain`
- `input_schema`
- `output_schema`
- `prompt_template` 或执行入口
- `guards`
- `defaults`
- `enabled`

### 5.2 Skill Registry

负责注册和查询技能定义。

职责：

- 加载技能定义
- 通过 `skill_id` 查找技能
- 提供按 domain / tag / capability 查询

### 5.3 Skill Executor

负责执行技能。

职责：

- 校验输入
- 构建 prompt / 调用 handler
- 应用 guard
- 返回标准结构化输出

### 5.4 Skill Router

负责在上层场景中选择技能。

职责：

- 根据页面动作、上下文、阶段选择 skill
- 例如：
  - 写第一章 -> `paper.chapter_draft`
  - 生成答辩大纲 -> `ppt.defense_outline`

### 5.5 Skill Guard

负责技能级约束。

职责：

- 限制引用规则
- 限制可用输入来源
- 限制输出字段
- 限制哪些章节允许 citations / 数据结果 / 风格选择

## 6. 与现有架构的关系

### 6.1 不替代 Agent

Skill 不是替代现有 Agent，而是：

- 给 Agent 外面再包一层可维护的定义和约束

### 6.2 不替代 Workflow

Skill 不替代现有 workflow runner，而是：

- Skill Executor 内部可以调用现有 workflow
- 或把一个 skill 映射为一个 workflow 节点组合

### 6.3 不直接绑定前端组件

前端只需要传：

- `skill_id`
- `context_id`
- `options`

避免前端直接知道过多 Agent 细节。

## 7. 第一阶段重点场景

本轮优先只覆盖两个领域：

### 7.1 论文写作 skill

建议拆分为：

- `paper.outline_generate`
- `paper.chapter_draft`
- `paper.chapter_grounding`
- `paper.references_collect`
- `paper.abstract_generate`

### 7.2 PPT 制作 skill

建议拆分为：

- `ppt.proposal_outline`
- `ppt.proposal_render`
- `ppt.defense_outline`
- `ppt.defense_render`
- `ppt.script_generate`

## 8. 目录结构建议

推荐新增：

```text
backend/app/skills/
├── registry.py
├── models.py
├── executor.py
├── router.py
├── guards.py
├── definitions/
│   ├── paper_outline_generate.py
│   ├── paper_chapter_draft.py
│   ├── paper_chapter_grounding.py
│   ├── paper_references_collect.py
│   ├── paper_abstract_generate.py
│   ├── ppt_proposal_outline.py
│   ├── ppt_proposal_render.py
│   ├── ppt_defense_outline.py
│   ├── ppt_defense_render.py
│   └── ppt_script_generate.py
└── __init__.py
```

## 9. 第一阶段最小实现原则

### 9.1 先文件注册，不先上数据库

第一阶段技能定义建议用 Python 文件或静态对象注册，不先引入数据库管理界面。

### 9.2 先覆盖写作和 PPT，不做全域平台

不要一开始就覆盖检索、设计、开题、写作、图谱全部场景。

### 9.3 先让 skill 成为“显式能力入口”

例如：

- 写作页点击 AI 续写 -> 调用 `paper.chapter_draft`
- 生成答辩 PPT -> 调用 `ppt.defense_render`

## 10. 当前最合适的实施顺序

建议顺序：

1. 新增 Skill Definition / Registry / Executor 最小骨架
2. 接入论文写作两个 skill：
   - `paper.chapter_draft`
   - `paper.chapter_grounding`
3. 接入 PPT 两个 skill：
   - `ppt.defense_outline`
   - `ppt.defense_render`
4. 再补 Router 和更细 skill 拆分

## 11. 不做事项

本方案当前不做：

1. 通用可视化技能平台
2. 用户可在线编辑 skill
3. 动态上传第三方 skill
4. 复杂 marketplace / 插件安装体系

## 12. 验收标准

第一阶段完成后，应达到：

1. 后端存在明确的 skill registry
2. 论文写作和 PPT 制作至少各有 2 个 skill 定义接入
3. skill 能通过统一 executor 调用
4. 现有页面至少有 1 到 2 个动作切换为 skill 调度
5. 不破坏当前 API 主链路

## 13. 需要确认

建议第一阶段只做“后端窄版技能系统”，不急着给前端暴露“技能选择器”。

也就是：

1. 后端先完成 skill 定义与执行层
2. 前端继续只触发原有按钮
3. 后端内部把按钮动作改成经由 skill 调度

如果确认继续，我建议下一步执行：

`阶段1：后端技能骨架最小实现`
