# 阶段4 01 验证与开发说明

## 目标

确保迁移体系不是“目录存在但没人用”，而是真正成为开发与部署流程的一部分。

## 需要补的说明

1. 本地新环境如何初始化数据库
2. 现有环境如何升级数据库
3. 新增字段/新表时的标准流程
4. 何时仍会触发 `schema_compat`
5. 何时应该使用 `stamp`

## 最小验证

1. Alembic 当前版本号可查询
2. 空库 upgrade 成功
3. 现有测试不因迁移接入而回归

## 推荐补充文档位置

1. `README.md`
2. `backend/README` 或 `docs/计划` 对应实施文档

## 推荐开发流程

### 新环境初始化

1. 安装依赖
2. 配置 `.env`
3. 执行 `python -m alembic -c alembic.ini upgrade head`
4. 启动后端

### 日常结构变更

1. 先修改 `app/models`
2. 执行 `python -m alembic -c alembic.ini revision -m "describe_change"`
3. 手工完善迁移文件
4. 执行 `python -m alembic -c alembic.ini upgrade head`
5. 跑测试

### 何时使用 stamp

只在以下场景使用：

1. 旧数据库已经有完整结构
2. 但没有 `alembic_version`
3. 且已经确认当前库结构与某个基线迁移一致

不建议在普通字段开发、普通环境升级中使用 `stamp`。
