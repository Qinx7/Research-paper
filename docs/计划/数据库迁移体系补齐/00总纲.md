# 数据库迁移体系补齐总纲

## 1. 目标

为当前项目补齐正式数据库迁移体系，结束仅依赖 `Base.metadata.create_all()` 与 `schema_compat` 运行时补丁的状态，建立一套可持续维护、可审计、可回放的结构变更机制。

## 2. 现状判断

当前后端数据库相关实现存在以下特点：

1. [database.py](/D:/learning%20agent/backend/app/core/database.py) 只负责 `SQLAlchemy` 连接与 `Base`
2. [main.py](/D:/learning%20agent/backend/app/main.py) 启动时直接执行 `Base.metadata.create_all(bind=engine)`
3. [schema_compat.py](/D:/learning%20agent/backend/app/services/schema_compat.py) 用运行时 SQL 做旧库补列

这意味着：

1. 新表/新字段的引入缺少显式迁移记录
2. 数据库结构变更无法被版本化审计
3. 启动阶段承担了“建表 + 兼容修补”的职责，容易掩盖结构问题
4. 生产环境和开发环境难以用同一种方式管理 schema

## 3. 目标边界

本轮做：

1. 引入 Alembic
2. 建立 `backend/alembic.ini`、`backend/alembic/` 迁移目录
3. 生成一个当前基线迁移
4. 约束后续字段/表变更通过迁移文件进入数据库
5. 保留 `schema_compat` 作为过渡兼容层，但降低其职责

本轮不做：

1. 一次性重写所有历史数据兼容逻辑
2. 自动把所有旧库不一致情况都无损修复
3. 上线复杂分支迁移治理

## 4. 关键假设

1. 当前项目继续使用 PostgreSQL
2. 当前数据库以“现有模型结构”作为迁移基线
3. 允许在过渡期内保留 `schema_compat`，但只用于旧环境兜底，不再承担长期演进职责
4. 本轮优先补“正式迁移机制”，而不是一次性清除所有历史技术债

## 5. 推荐策略

### 5.1 总体策略

采用“**先建立基线迁移，再收缩运行时兼容补丁**”的方式：

1. 先把 Alembic 框架搭起来
2. 生成一版当前 schema 的基线迁移
3. 调整启动逻辑，逐步弱化 `create_all`
4. 后续所有结构变更走 Alembic revision

### 5.2 过渡策略

过渡期内：

1. 开发和新环境优先使用 Alembic 升级数据库
2. `schema_compat` 暂时保留，用于旧库缺少关键列时兜底
3. `create_all` 不应继续作为长期主路径

## 6. 风险

1. 如果直接移除 `create_all`，当前旧库或空库可能启动失败
2. 如果 Alembic 基线与现有实际库结构不一致，可能产生假迁移或冲突
3. 如果不明确过渡策略，团队会继续绕过迁移体系直接改模型

## 7. 阶段拆分

### 阶段1：Alembic 骨架搭建

目标：

1. 增加 Alembic 配置与目录
2. 让迁移框架能读取当前 SQLAlchemy metadata

### 阶段2：当前 schema 基线迁移

目标：

1. 生成一版初始迁移
2. 以当前已落地模型作为迁移起点

### 阶段3：启动路径调整

目标：

1. 弱化 `create_all`
2. 保留最小兼容兜底
3. 明确“数据库升级靠 Alembic”

### 阶段4：验证与文档

目标：

1. 验证空库初始化
2. 验证旧库兼容
3. 补开发说明

## 8. 成功标准

1. 项目中存在正式 Alembic 迁移目录与配置
2. 当前 schema 有基线迁移文件
3. 新环境可通过 Alembic 初始化数据库
4. 后续结构变更不再依赖 `create_all`
5. 启动阶段不再承担长期 schema 演进职责
