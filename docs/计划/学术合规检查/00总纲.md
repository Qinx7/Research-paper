# 学术合规检查方案总纲

## 1. 需求理解

当前系统在论文生成阶段已通过 LLM 提示词约束学术合规行为（`CHAPTER_SYSTEM_PROMPT` 明确禁止编造数据/引用），但**没有任何验证机制**。LLM 仍可能：

- 在无真实数据时生成"实验结果表明…准确率达到 95.3%"
- 编造不存在的文献引用
- 声称引用了某成果但实际未上传
- `data_based` 字段与实际内容不一致

本次目标：在 LLM 生成章节后，增加一套**事后验证**机制，自动检测上述违规行为。

## 2. 关键设计决策（已确认）

### 决策 1：规则引擎 + AI 深度检查，AI 必须接地

- **规则检查**（确定性）：正则匹配、关键词扫描、数据库比对。覆盖 80% 的违规模式，零延迟、可解释。自动执行。
- **AI 深度检查**：对模糊边界（如措辞过度自信、隐含的数据断言）调用 LLM 做语义判断。**关键约束**：AI 只能做"对比分析"——将章节文本与已有成果/文献进行匹配验证，不得生成任何新内容或推断未见数据。提示词设计为纯验证角色，输入仅为 (章节文本, 成果列表, 文献列表)，输出仅为 issues 清单。

### 决策 2：作为独立服务模块，不嵌入 Agent

新建 `backend/app/services/compliance_checker.py`，与现有 Agent 解耦。Agent 负责生成，ComplianceChecker 负责验证。

### 决策 3：自动检查 + 用户确认，不阻断

流程：章节生成后自动运行合规检查 → 前端展示 issues 列表 → **用户逐条审核确认**（通过/忽略/修正）→ 确认后的合规状态持久化。检查结果不自动阻止编辑和导出，但用户确认前的章节标记为"待审核"。

## 3. 检查项定义（5 项）

| # | 检查项 | 类型 | 触发条件 | 方法 |
|---|--------|------|----------|------|
| 1 | **数据真实性** | error | 章节未标记 `data_based` 但出现"实验结果表明""数据显示"等已完成措辞 | 正则匹配违禁词列表 |
| 2 | **引用真实性** | error | 正文中引用的文献编号不在 `draft.references` 中，或参考文献包含 LLM 编造特征（假 DOI、格式异常） | 提取引用编号 → 比对数据库 → DOI 格式校验 |
| 3 | **章节标记** | warning | 第 4、5 章缺少 `[基于真实数据]` 或 `[实验设计方案]` 标记 | 关键词匹配 |
| 4 | **统计数据** | warning | 无实验数据成果时，正文出现"达到XX%""F1值为XX"等数值断言 | 正则匹配数值断言模式 + 交叉验证 outcome 列表 |
| 5 | **AI 语义对比** | info | 规则检查完成后，调用 LLM 做接地对比分析 | 纯验证角色 prompt，只输出不匹配项 |

### AI 语义对比的接地约束

AI 检查的 system prompt 设计原则：
- **角色**：学术合规审计员，不是论文写作者
- **输入**：仅限 (当前章节全文, 项目成果列表, draft.references 列表)
- **输出**：仅限 `[{issue_type, location, description, evidence}]` 格式的 issues 清单
- **禁止行为**：不得推断数据、不得补充结论、不得改写内容、不得建议替代措辞
- **空结果处理**：如果未发现合规问题，返回空清单 `[]`，不生成任何解释

## 4. 实施阶段

### 阶段 1：合规检查服务 + Schema

**步骤 1.1：新建 Schema**
- 新建 `backend/app/schemas/compliance.py`
- 定义 `ComplianceIssue`、`ChapterCompliance`、`ComplianceResult`、`ComplianceConfirmRequest`

**步骤 1.2：新建 ComplianceChecker 服务**
- 新建 `backend/app/services/compliance_checker.py`
- 实现 4 项规则检查函数 + 1 项 AI 接地对比函数
- 主入口：`check_draft(draft, outcomes, papers, enable_ai=True) → ComplianceResult`

### 阶段 2：API + 自动集成

**步骤 2.1：新增 API 端点**
- 修改 `backend/app/api/drafts.py`
- `POST /api/drafts/{id}/check-compliance` — 运行检查（自动包含规则 + AI）
- `POST /api/drafts/{id}/confirm-compliance` — 用户逐条确认 issues
- `GET /api/drafts/{id}/compliance-status` — 获取当前合规状态

**步骤 2.2：章节生成后自动检查**
- 修改 `backend/app/tasks/paper_task.py`
- `generate_chapter_task` 完成后自动运行 4 项规则检查（不跑 AI，避免阻塞 Celery 任务）
- AI 深度检查由用户在前端手动触发

### 阶段 3：前端展示 + 用户确认

**步骤 3.1：合规面板组件**
- 在 `PaperWorkflow.tsx` 章节编辑器中增加合规检查面板
- 自动显示规则检查结果（error/warning）
- "深度检查"按钮触发 AI 语义对比
- 每条 issue 可点击"确认通过"/"标记忽略"/"去修改"

## 6. 验证方式

1. 无成果时生成第 5 章 → 规则检查应检测到 `data_based=false` 内容中的违禁措辞
2. 编造引用编号 → 应标记 `fake_reference` error
3. AI 深度检查 → 输入仅限已有数据，不应输出任何推断性内容
4. 前端 → issues 列表展示 + 确认/忽略按钮可用
