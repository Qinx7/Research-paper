# 论文写作 Skill 自研方案总纲

## 1. 需求理解

当前项目已经有论文写作相关能力，但现状是：

- 后端 `skill runtime` 已有最小骨架。
- 论文单章节生成与依据校验已经通过 `paper.chapter_draft`、`paper.chapter_grounding` 接入。
- 论文写作能力还没有形成一套完整、可扩展、可开源的 `paper.*` 技能体系。

本方案目标不是直接复用外部 `academic-research-skills` 仓库内容，而是参考其流程设计，自研一套兼容当前项目的论文写作技能系统。

## 2. 关键约束

### 2.1 开源许可约束

- 外部仓库 `Imbad0202/academic-research-skills` 使用 `CC-BY-NC-4.0`，不适合直接并入我们后续要开源的项目。
- 本项目后续要面向开源发布，因此不能直接复制该仓库的 `SKILL.md`、提示词文本、流程文案或资源文件。
- 可以参考其方法论，但必须重新设计、重新表述、重新实现。

### 2.2 当前系统约束

- 首页只做文献检索，不新增新的首页长链路入口。
- 论文写作能力仍以当前写作页和项目页入口为主，不额外引入 CLI 风格交互。
- 不新增重型依赖，优先复用现有 `SkillRuntime / AgentWorkflow / grounding_guard / evidence retrieval`。
- 与现有数据结构兼容，优先复用 `draft / outcomes / papers / evidence_items`。

## 3. 当前已具备基础

当前已存在可复用底座：

- `SkillRegistry / SkillExecutor / SkillRouter / SkillRuntime`
- `paper.chapter_draft`
- `paper.chapter_grounding`
- 论文章节写作 workflow
- 证据检索与引用依据校验

这意味着后续工作重点不是“从零搭建技能系统”，而是把论文写作能力继续拆解成更清晰的 `paper.*` 技能，并补足页面与流程接入。

## 4. 自研 Skill 目标结构

建议把论文写作能力收敛成 6 类技能：

1. `paper.plan`
2. `paper.outline_generate`
3. `paper.section_draft`
4. `paper.citation_verify`
5. `paper.review_pass`
6. `paper.revision_apply`

其中：

- `paper.section_draft` 可以兼容或逐步替代当前 `paper.chapter_draft`
- `paper.citation_verify` 可以兼容或逐步替代当前 `paper.chapter_grounding`

## 5. 分阶段建议

### 阶段1：能力模型收口

目标：

- 定清楚论文写作 skill 的边界、命名、输入输出和与现有 workflow 的映射关系。
- 不急着扩实现，先避免技能名、职责和页面动作继续散掉。

### 阶段2：最小可用接入

目标：

- 在保留当前可用功能的前提下，引入 `paper.plan`、`paper.outline_generate`。
- 让“生成大纲、章节写作、依据校验”形成连续 skill 链。

### 阶段3：审稿与修订闭环

目标：

- 补上 `paper.review_pass`、`paper.revision_apply`。
- 让“生成后检查、指出问题、应用修订”成为产品闭环，而不只是单次生成。

### 阶段4：页面承接与产品化

目标：

- 把 skill 状态和动作收敛到现有写作页。
- 不暴露开发者视角的执行细节，只保留用户可理解的写作动作。

## 6. 设计原则

1. 不复制外部仓库文本，只借思路。
2. 不引入另一套独立 orchestration，统一走当前后端 skill runtime。
3. 每个 skill 必须有明确输入、输出、约束和失败语义。
4. 每个 skill 尽量对应一个用户可理解动作，避免纯技术拆分。
5. 先补产品闭环，再追求 agent 数量和复杂编排。

## 7. 本轮建议确认事项

本轮只建议先做论文写作 skill 方案，不直接进入代码实现。

后续如果开始实施，优先顺序建议是：

1. `paper.plan`
2. `paper.outline_generate`
3. 统一 `paper.chapter_draft / paper.chapter_grounding` 命名
4. `paper.review_pass`
5. `paper.revision_apply`
