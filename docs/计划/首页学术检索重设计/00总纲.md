# 首页学术检索重设计总纲

## 需求理解

用户希望参考 PaSa Agent、OpenAlex、Consensus 三类学术检索产品的模式，为本项目首页重新设计一种更清晰、更有辨识度的检索方式。目标不是照搬三个网站外观，而是吸收它们的检索逻辑，形成适合“文献驱动型研究生科研 Agent”的首页入口。

## 参考模式提炼

### PaSa：代理式检索

PaSa 的核心不是普通关键词搜索，而是让代理围绕复杂学术问题自主执行多步任务：改写查询、调用检索工具、阅读论文、筛选相关参考文献。可借鉴点是“检索过程可被编排”，首页应让用户看到系统会经历“理解问题 → 生成检索策略 → 检索文献 → 评估证据 → 进入分析”的链路。

### OpenAlex：开放知识图谱与分面探索

OpenAlex 的价值在于开放学术实体网络，支持 works、authors、sources、institutions、topics 等实体，并提供搜索、语义搜索、自动补全、过滤和 group_by 聚合能力。可借鉴点是“检索不是一列结果，而是一个可探索的证据空间”，首页应强调关键词、实体、年份、来源、语言、影响力等过滤维度。

### Consensus：答案优先与证据共识

Consensus 的首页强调 “Research starts here”，入口围绕 Sources、Corpus、Deep、Filter 等清晰动作展开。可借鉴点是“用户先提出问题，系统给出可追溯答案和证据层级”，首页应把检索目标表达成“我要得到什么类型的研究判断”，而不仅是“我要搜什么词”。

## 我们的独特风格：证据罗盘

建议将首页检索设计为“证据罗盘 Evidence Compass”：

- 中央是研究问题输入框，支持自然语言描述研究方向。
- 左侧或顶部是检索意图卡片：快速摸底、综述梳理、深度研究。
- 右侧是证据策略面板：中文/英文/全部，年份范围，最低引用，高影响优先。
- 底部展示系统将执行的研究路径：问题理解、关键词抽取、跨库检索、证据评分、研究方向生成。
- 结果区以“证据卡片 + 来源分布 + 下一步按钮”呈现，而不是单纯列表。

## 当前项目落点

当前首页实际由 `frontend/src/components/PipelineWizard.tsx` 承载，流程为：

1. 第 1 步：输入研究需求并调用 `api.analyzeRequirement`。
2. 第 2 步：根据关键词调用 `api.searchLiterature`。
3. 第 3 步：调用 `api.analyzeLiterature`。
4. 后续生成研究方向、项目设计、开题成果。

因此第一轮改造应优先重设计 `PipelineWizard` 的首页与第 1、2 步视觉/交互，不改后端接口、不新增依赖、不改变现有数据结构。

## 实施边界

本轮建议只做前端交互与样式重组：

- 保留现有 `ResearchMode`：`quick_search`、`literature_review`、`deep_research`。
- 保留现有 `LibraryScope`：`all`、`cn`、`en`。
- 保留现有 API 调用链路。
- 不新增外部搜索 API。
- 不改变数据库结构。
- 不引入新的 UI 依赖。

## 验收标准

- 首页首屏能明显看出这是一个学术检索入口，而不是普通表单。
- 用户能在开始前选择检索模式、文献范围和基础筛选策略。
- 第 1 步需求分析结果能以“检索策略预览”的方式呈现。
- 第 2 步文献检索结果能显示来源分布、数量和证据卡片。
- 不影响后续文献分析、研究方向、项目设计和开题材料生成。
- `npm.cmd run build` 通过。

## 需要用户确认

建议确认首页检索体验采用以下方向：

- 方案名称：证据罗盘 Evidence Compass。
- 改造范围：优先改 `PipelineWizard` 的首屏、第 1 步和第 2 步。
- 后端策略：本轮不改后端，仅复用已有检索模式和参数。
- 视觉方向：延续当前深绿、暖纸、学术编辑风格，但让检索入口更像专业研究仪表盘。
