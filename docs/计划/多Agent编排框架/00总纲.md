# 多 Agent 编排框架总纲

## 1. 需求理解

当前项目已经有多个独立 Agent：需求理解、文献检索、文献分析、研究方向、项目设计、论文写作、PPT、答辩等。但这些 Agent 目前主要由 API 路由手动串联，缺少统一的任务状态、上下文传递、节点执行记录、失败恢复、证据约束和可观测能力。

本方案优先实现一个轻量、可落地的多 Agent 编排框架，让现有 Agent 可以逐步接入，而不是一次性重写。第一阶段目标是跑通一个真实主路径：首页文献检索，形成“查询理解 → 来源选择 → 多源检索 → 结果归一化 → 质量排序 → 本次检索总结分析 → 任务记录”的可追踪流程。

## 2. 当前现状

已存在能力：

- `backend/app/agents/` 下已有多个单体 Agent。
- 首页已经作为文献检索入口，检索能力主要由 `literature_search_agent` 和 `literature_search_tasks` 承接。
- 论文写作已经支持大纲、章节、摘要生成，并有基础 grounding guard。
- 检索任务已有 `literature_search_tasks` 记录。
- 异步任务已有 Celery/Redis 基础，但主要用于章节/PPT/报告生成。

主要缺口：

- 没有统一的 Agent 节点接口。
- 没有 workflow/run/step 级执行记录。
- 没有统一状态机：pending/running/success/failed/skipped/retrying。
- 没有跨 Agent 的上下文对象，现有参数传递分散在 API 中。
- 没有统一错误处理、重试策略和降级策略。
- 没有统一的“检索输入/来源状态/结果质量/本次检索分析/诊断输出”协议，容易出现空结果、来源失败不可见、排序不可解释、检索结果缺少概览。

## 3. 总体目标

新增一个轻量多 Agent 编排层，满足：

- 可以定义一个 workflow，由多个 agent node 串行或条件分支执行。
- 每个 node 输入和输出都结构化记录。
- workflow 运行过程可追踪、可恢复、可诊断。
- 支持同步运行和后续异步运行扩展。
- 优先复用现有 Agent，不大规模重构业务逻辑。
- 最先接入首页文献检索路径，减少空结果、来源状态不透明、中文/英文来源失衡、排序不可解释和结果缺少总结分析的问题。

## 4. 推荐方案

推荐采用“自研轻量编排层”，暂不直接引入 LangGraph。

理由：

- 当前项目已有多个同步 Python Agent 和 FastAPI 路由，直接引入 LangGraph 会带来额外依赖和改造成本。
- 现阶段最需要的是统一状态、上下文、日志和执行记录，而不是复杂图运行时。
- 轻量编排层可以先用普通 Python 类实现，后续如果确实需要 LangGraph，再把 node 接口迁移过去。

## 5. 可选方案对比

### 方案 A：自研轻量编排层（推荐）

优点：

- 改动小，能快速跑通。
- 不新增重依赖。
- 可按现有项目风格写单元测试。
- 更容易先接入现有 `literature_search_agent.py` 和 `literature_search_tasks.py`，后续再扩展到论文写作等路径。

缺点：

- 不具备 LangGraph 现成生态。
- 后续复杂分支和人工介入需要自己扩展。

### 方案 B：直接引入 LangGraph

优点：

- 状态图、条件边、节点编排能力成熟。
- 后续复杂多 Agent 流程更标准。

缺点：

- 新增依赖和学习成本。
- 现有代码需要更大改造。
- 当前项目还没有清晰的统一 agent state，贸然引入容易变成“套壳框架”。

### 方案 C：只在文献检索接口里继续手写串联

优点：

- 最快，短期改动最小。

缺点：

- 继续扩大 `literature.py` / `literature_search_agent.py` 的复杂度。
- 不能复用到论文写作、开题、PPT。
- 后续调试和失败恢复仍然困难。

## 6. 建议优先落点

第一条接入主路径：首页文献检索 workflow。

原因：

- 首页只做文献检索，这是当前产品边界。
- 当前检索已经串了多个步骤：查询理解、关键词、中英文范围、多源检索、来源状态、去重、排序、结果总结、任务记录。
- 多 Agent 编排能直接改善“空结果、来源不清、中文文献缺失、限流诊断不可见”的问题。

首个 workflow：

```mermaid
flowchart LR
    A["Search Query"] --> B["Query Planner Agent"]
    B --> C["Source Router Agent"]
    C --> D["Multi-source Search Agent"]
    D --> E["Result Normalizer Agent"]
    E --> F["Quality Ranker Agent"]
    F --> G["Search Synthesis Agent"]
    G --> H["Search Diagnostics Agent"]
    H --> I["Search Results + Summary"]
```

## 7. 分阶段实施

### 阶段 1：编排框架骨架

新增 workflow run、node、state、event、runner 的最小实现。先不接 UI，只用单元测试证明节点可以顺序执行、失败停止、输出可追踪。

### 阶段 2：首页文献检索 workflow 接入

把当前文献检索中的查询规划、来源选择、多源检索、结果归一化、质量排序、本次检索总结分析、来源诊断封装为 node。首页仍然只做文献检索，不生成连续对话回答。

### 阶段 3：执行记录和诊断

新增 `agent_workflow_runs` 与 `agent_workflow_steps` 表，记录每次 workflow 的输入、输出摘要、状态、错误、耗时和关联 conversation/project/task。

### 阶段 4：论文写作 workflow 接入

把论文大纲、章节生成、证据检索、grounding guard 接入 workflow。目标是减少论文生成幻觉，并让每个章节知道自己依据哪些资料。

### 阶段 5：开题主流程接入

后续再接需求理解、文献分析、研究方向、项目设计、开题报告、PPT。这个阶段暂不优先，因为你前面明确过开题完整工作流不是当前首要目标。

## 8. 验收标准

- 后端有统一 workflow runner 和 node 接口。
- 首页文献检索能通过 workflow 执行，但前端调用方式不变。
- 每个节点有明确状态和错误信息。
- 出错时返回明确来源诊断，不会只表现为空结果。
- 查询规划、来源选择、检索结果、排序、总结分析、诊断之间的数据传递可追踪。
- 单元测试覆盖：顺序执行、节点失败、条件跳过、上下文输出合并。
- 不新增大型 UI，不改变首页只做文献检索的主交互。

## 9. 不做范围

第一轮不做：

- 完整 LangGraph 引入。
- 可视化拖拽 workflow。
- 连续聊天式问答。
- 多用户协同审批。
- 全部 Agent 一次性迁移。
- 复杂人工中断/恢复。
- 分布式 worker 编排。

## 10. 需要确认

- 第一阶段是否按“自研轻量编排层”实现。
- 首个落点是否确定为“首页文献检索 workflow”。
- 是否允许新增两张表：`agent_workflow_runs`、`agent_workflow_steps`。
