# 阶段 1：编排框架骨架

## 1. 目标

先实现一个不依赖具体业务的轻量 workflow runner。它只负责执行节点、维护上下文、记录状态和处理错误。这个阶段不改前端，不改变已有 API 行为。

## 2. 建议新增文件

- `backend/app/agents/orchestration/__init__.py`
- `backend/app/agents/orchestration/state.py`
- `backend/app/agents/orchestration/node.py`
- `backend/app/agents/orchestration/runner.py`
- `backend/app/agents/orchestration/events.py`
- `backend/tests/test_agent_workflow_runner.py`

## 3. 核心对象

### AgentWorkflowState

统一上下文对象，建议包含：

- `run_id`
- `workflow_name`
- `user_id`
- `project_id`
- `conversation_id`
- `input`
- `data`
- `evidence`
- `messages`
- `errors`
- `metadata`

### AgentNode

统一节点接口：

- `name`
- `description`
- `run(state) -> AgentNodeResult`
- `should_run(state) -> bool`

### AgentNodeResult

节点输出：

- `status`: `success | failed | skipped`
- `data_delta`
- `evidence_delta`
- `messages`
- `error`
- `metadata`

### AgentWorkflowRunner

执行器：

- 顺序执行节点。
- 支持节点跳过。
- 节点失败时停止后续关键节点。
- 将每个节点输出合并到 state。
- 返回最终 state 和事件列表。

## 4. 最小测试

需要先写测试：

- 两个节点顺序执行，第二个节点能读取第一个节点输出。
- `should_run=False` 时节点状态为 skipped。
- 节点抛异常时 runner 捕获错误并停止。
- 节点返回 `data_delta` 后合并到 state。

## 5. 验收标准

- runner 单元测试通过。
- 没有引入业务 Agent。
- 没有新增外部依赖。
- 后端 compileall 通过。
