# 阶段 3：执行记录与诊断

## 1. 目标

让每次 workflow 执行可追踪。用户或开发者可以知道哪一步失败、耗时多少、使用了哪些检索参数、命中了哪些来源、最终返回了多少文献，以及本次检索总结是否成功生成。

## 2. 建议新增模型

### AgentWorkflowRun

字段建议：

- `id`
- `workflow_name`
- `status`
- `user_id`
- `project_id`
- `search_task_id`
- `input_snapshot`
- `output_snapshot`
- `error_message`
- `started_at`
- `finished_at`
- `created_at`

### AgentWorkflowStep

字段建议：

- `id`
- `run_id`
- `node_name`
- `status`
- `input_summary`
- `output_summary`
- `error_message`
- `duration_ms`
- `started_at`
- `finished_at`

## 3. API 建议

先只做后端诊断接口：

- `GET /api/agent-workflows/runs`
- `GET /api/agent-workflows/runs/{run_id}`

第一轮可以不做前端页面。

## 4. 数据原则

- 不保存完整超长查询扩展内容。
- 保存摘要、计数、状态、错误。
- 对检索结果只保存数量、来源分布、前若干标题摘要和总结摘要，避免日志过大。
- 和用户/项目权限绑定。

## 5. 验收标准

- 一次首页文献检索 workflow 能产生 run 和 step 记录。
- 失败节点能记录错误。
- 总结分析节点能记录“已生成 / 依据不足 / 生成失败”状态。
- 查询接口只能看到当前用户自己的 run。
- 不影响原有首页检索响应速度太多。
