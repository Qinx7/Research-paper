# 整篇审查整篇修订 总纲

## 1. 需求理解

当前项目已经完成：

- `/writing` 作为整篇论文同页生成与编辑入口
- 整篇初稿生成
- 当前章节审查
- 当前章节修订

但还缺少真正面向“整篇论文”的两类能力：

- 整篇审查
- 整篇修订

也就是说，现在系统仍然偏“整篇编辑 + 单章节增强”，还不是“整篇论文工作台闭环”。

## 2. 本轮目标

本轮目标是补齐两条主链路：

1. 整篇审查
2. 整篇修订

并且要求：

- 继续留在 `/writing` 同一页完成
- 不新增新页面
- 不改数据库结构
- 优先复用现有 `draft`、`paper_writing_agent`、`skill runtime`

## 3. 目标效果

用户在 `/writing` 中应能做到：

1. 生成完整初稿
2. 查看整篇连续正文
3. 发起整篇审查
4. 得到整篇问题摘要与问题清单
5. 发起整篇修订
6. 得到整篇修订摘要、已解决问题、剩余问题
7. 继续在同页整篇人工编辑

## 4. 范围边界

### 本轮要做

- 后端整篇审查能力
- 后端整篇修订能力
- `/writing` 页面整篇审查与整篇修订入口
- 审查结果、修订结果、剩余问题的整篇展示

### 本轮不做

- 多版本历史回滚
- 段落级富 diff
- 协同审阅
- 审查任务异步化
- 数据库新表

## 5. 建议能力拆分

### 5.1 后端 skill

- `paper.full_review_pass`
- `paper.full_revision_apply`

### 5.2 后端接口

- `POST /api/drafts/{draft_id}/review-full`
- `POST /api/drafts/{draft_id}/revise-full`

### 5.3 前端 `/writing`

- 工具栏新增：
  - `整篇审查`
  - `整篇修订`
- 右侧面板新增：
  - `整篇审查结果`
  - `整篇修订结果`
  - `整篇修订状态`

## 6. 关键设计原则

1. 整篇能力优先，不再继续围绕单章节堆功能。
2. 整篇结果应以“全文正文”作为输入，不以单章节状态拼凑替代。
3. 整篇修订要优先使用 LLM，失败时回退到规则型修订。
4. 整篇审查结果要区分：
   - 结构问题
   - 证据问题
   - 重复问题
   - 术语一致性问题
   - 摘要/结论呼应问题
5. 所有动作都留在 `/writing` 同一页。

## 7. 分阶段建议

- 阶段1：后端整篇审查最小实现
- 阶段2：后端整篇修订最小实现
- 阶段3：`/writing` 页面整篇操作入口与展示
- 阶段4：整篇审查/修订结果增强与回归验证
