# 阶段2 01 整篇修订最小实现

## 1. 目标

在整篇审查输出的基础上，提供最小整篇修订能力。

## 2. 后端建议

新增：

- `paper.full_revision_apply`

输入：

- `draft_title`
- `full_text`
- `issues`
- `focus_areas`
- `citations`
- `evidence_context`

输出：

- `title`
- `full_text`
- `change_summary`
- `resolved_issues`
- `remaining_issues`

## 3. 修订策略

### 优先路径

- 有 LLM key 时，整篇修订优先走 LLM

### fallback 路径

- 无 key 或修订失败时：
  - 只做保守的规则型修订
  - 不做大规模重写
  - 重点去掉明显风险表述、补充承接句、统一术语

## 4. 接口

- `POST /api/drafts/{draft_id}/revise-full`

## 5. 风险点

1. 整篇重写容易覆盖用户手工修改
2. 长文本修订容易超 token
3. 整篇修订结果需要明确标记为“已修订版本”

## 6. 本阶段建议

本阶段先做：

- 同步接口
- 覆盖整篇正文
- 更新草稿版本号

暂不做：

- 多版本 diff
- 回滚历史
