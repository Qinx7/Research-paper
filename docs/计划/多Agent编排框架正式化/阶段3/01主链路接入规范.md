# 阶段 3：主链路 workflow 接入规范

## 1. 目标

把当前主链路统一成可复用的 workflow facade，减少各 API 自己拼状态、自己判断是否重跑、自己写结果的分散逻辑。

## 2. 接入范围

当前优先覆盖：

1. 首页文献检索与本次检索总结。
2. 研究方向生成与评分。
3. 项目设计生成与保存。
4. 论文写作章节生成、整篇审查、整篇修订。
5. 通用 PPT / HTML Deck 生成。

不覆盖：

1. 开题报告。
2. 答辩问题预测。
3. 已下线的 pipeline 页面。
4. 普通用户执行记录页面。

## 3. Workflow Facade 规范

每条主链路建议暴露一个明确的 facade 函数：

- `run_home_literature_search_workflow`
- `run_research_direction_workflow`
- `run_project_design_workflow`
- `run_paper_chapter_generation_workflow`
- `run_paper_full_review_workflow`
- `run_paper_full_revision_workflow`
- `run_deck_generation_workflow`

Facade 负责：

1. 接收 API 层参数。
2. 判断是否可以复用历史结果。
3. 创建 `AgentWorkflowState`。
4. 注入 recorder、skill runtime、业务服务。
5. 运行 runner。
6. 返回原 API 兼容结构。

## 4. 历史结果复用规范

以下场景应该优先复用历史结果，而不是重新生成：

1. 首页点击历史检索记录。
2. 研究页重新打开已有需求理解、文献分析、研究方向。
3. 重新打开已保存项目设计。
4. 写作页查看已生成章节。
5. 打开已生成 HTML Deck 预览。

以下场景才重新运行 workflow：

1. 用户明确点击“重新生成”。
2. 输入内容变化导致 `input_hash` 不一致。
3. 历史结果不存在或已损坏。
4. workflow 版本升级后被标记为不可复用。

## 5. 结果引用规范

每次 workflow 应在 `result_ref` 中记录可恢复结果：

首页检索：

- `search_task_id`
- `query`
- `paper_count`

研究方向：

- `project_id`
- `direction_ids`

项目设计：

- `project_id`
- `design_id`

论文写作：

- `draft_id`
- `chapter_key`
- `draft_version`

PPT / Deck：

- `artifact_id`
- `object_key`
- `preview_url`
- `download_url`

## 6. 失败处理规范

主链路失败分为三类：

1. `failed`：关键节点失败，不能产出结果。
2. `partial_success`：主结果已生成，但部分来源或辅助诊断失败。
3. `success_with_warnings`：结果可用，但存在非阻塞警告。

前端产品页只需要展示用户可理解的提示，不展示内部节点名和堆栈。

## 7. 验收标准

1. 现有 API 返回结构兼容。
2. 历史记录打开不触发重新生成。
3. 用户明确重新生成时才创建新的 workflow run。
4. 每条主链路都能写入 `result_ref`。
5. 失败场景返回用户可读错误，内部诊断写入 workflow 记录。
