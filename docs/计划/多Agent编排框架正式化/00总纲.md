# P1-1 多 Agent 编排框架正式化方案

## 1. 需求理解

当前项目已经不是“没有多 Agent”，而是已经形成了三层雏形：

1. `backend/app/agents/orchestration/`：轻量 workflow runner、节点、状态、事件。
2. `backend/app/agents/workflows/`：首页检索、研究方向、项目设计、论文写作等业务 workflow。
3. `backend/app/skills/`：论文、研究、PPT 的 skill registry、router、executor。

P1-1 的目标不是推翻现有实现，也不是马上引入 LangGraph 这类外部框架，而是把现有自研 `workflow + skill runtime` 正式化：形成稳定契约、统一事件、可诊断记录、可复用节点规范，并让后续检索、研究、写作、PPT 都能按同一套方式扩展。

## 2. 当前问题

### 2.1 编排层有骨架，但契约还不够稳定

- `AgentWorkflowState`、`AgentNode`、`AgentNodeResult` 已存在。
- 节点输入输出主要靠约定，缺少明确的节点类型、输入摘要、输出摘要、证据引用、skill 调用记录规范。
- 不同 workflow 自己拼装 state 和 metadata，后续扩展容易出现字段不一致。

### 2.2 skill runtime 已接入部分链路，但还不是统一调用面

- 写作、研究方向、项目设计已经通过 `SkillRouter + SkillExecutor` 调用。
- 首页检索仍主要直接调用文献检索 Agent。
- PPT 已有 `ppt.web_html_deck` skill，但与 workflow 编排的统一关系还需要明确。

### 2.3 执行记录存在，但更偏开发记录

- `agent_workflow_runs` 与 `agent_workflow_steps` 已经能记录运行和节点。
- 记录内容还没有形成“用户可见摘要”和“内部诊断详情”的分层。
- 用户此前已明确：产品前端不需要展示执行记录按钮。因此本方案只保留内部诊断能力，不把它做成主产品入口。

### 2.4 主链路恢复与重放边界需要更清楚

- 首页历史检索、研究方向、项目设计、写作生成都涉及“已有结果再次打开不要重复生成”。
- 多 Agent 编排层应记录结果引用、输入摘要和产物引用，帮助页面恢复结果，而不是每次重新跑生成。

## 3. 推荐路线

推荐采用“自研轻量正式化”路线：

1. 保留现有 `AgentWorkflowRunner`，不引入新依赖。
2. 先补契约、事件、记录规范，再逐步改造业务 workflow。
3. 将 skill runtime 作为节点能力调用层，而不是另起一套 Agent 框架。
4. 内部诊断留在后端 API 和开发文档，不在产品首页或研究页暴露“执行记录”入口。

## 4. 不推荐路线

### 4.1 直接引入 LangGraph

优点是框架成熟，有图式编排能力。缺点是当前项目主链路已经在跑，引入新依赖和新抽象会带来迁移成本，且短期不能解决历史恢复、证据记录、skill 调用一致性这些更具体的问题。

### 4.2 只写文档不改结构

优点是成本最低。缺点是无法约束后续实现，节点字段和记录结构仍会继续漂移，P1-1 的价值不够。

## 5. 正式化后的目标结构

```text
业务入口 API
  ↓
Workflow Facade
  - 负责参数兼容
  - 负责是否复用历史结果
  - 负责创建统一 state
  ↓
AgentWorkflowRunner
  - 顺序执行节点
  - 维护状态、事件和失败策略
  - 调用 recorder
  ↓
AgentNode
  - context / retrieve / generate / validate / persist / render
  - 节点通过 SkillRuntime 或原有 Agent 服务完成实际能力
  ↓
SkillRuntime / Domain Agent / Service
  - 论文写作 skill
  - 研究 skill
  - PPT skill
  - 检索、证据、知识库等服务
```

## 6. 产品边界

本方案继续遵守当前 PRD v2 产品边界：

1. 首页只做文献检索，以及本次检索内容的总结分析。
2. 研究页承接研究方向和项目设计。
3. 写作页承接整篇论文生成、编辑、审查、修订。
4. 通用 PPT / HTML Deck 保留，但不恢复开题报告或答辩相关链路。
5. `/workflows` 不作为用户产品页出现，内部诊断不放到普通用户主入口。

## 7. 分阶段实施

### 阶段 1：编排契约最小正式化

目标是先让 runner、state、node、result、event 有稳定字段和测试，不改变现有 API 返回结构。

文档：`阶段1/01编排契约最小正式化.md`

### 阶段 2：执行记录与诊断标准化

目标是让 run / step 记录可以稳定表达节点类型、skill 调用、证据数量、结果引用、失败原因，并区分用户摘要和内部诊断。

文档：`阶段2/01执行记录与诊断标准化.md`

### 阶段 3：主链路 workflow 接入规范

目标是把首页检索、研究方向、项目设计、论文写作、PPT/Deck 统一成同一套 workflow facade 规范，并明确哪些场景复用历史结果。

文档：`阶段3/01主链路接入规范.md`

### 阶段 4：skill runtime 与节点能力收敛

目标是让生成类、校验类、渲染类能力都通过稳定 skill 调用面进入 workflow，减少业务 workflow 直接散落调用 Agent 的情况。

文档：`阶段4/01Skill调用面收敛.md`

### 阶段 5：验收与回归

目标是用最小但关键的测试覆盖 runner、recorder、skill 调用、历史结果恢复和主链路 API 兼容性。

文档：`阶段5/01验收与回归清单.md`

## 8. 关键假设

1. 本轮 P1-1 不新增外部依赖。
2. 本轮不恢复开题报告、答辩问题预测或已下线页面。
3. 内部诊断可以通过后端 API 保留，但不在普通用户前端暴露。
4. 数据库字段如需扩展，走 Alembic 迁移，不再依赖 `create_all` 隐式补表。
5. 阶段 1 只做最小契约正式化，不一次性重写全部 workflow。

## 9. 需要确认

建议先执行阶段 1。阶段 1 的改动低风险，主要围绕编排层契约和测试，不触碰页面体验和主业务逻辑。
