# 阶段 1：编排契约最小正式化

## 1. 目标

在不改变现有业务 API 行为的前提下，正式定义多 Agent workflow 的最小契约。阶段 1 只做“让编排层有稳定语言”的工作，不迁移全部业务链路。

## 2. 当前基础

已有文件：

- `backend/app/agents/orchestration/state.py`
- `backend/app/agents/orchestration/node.py`
- `backend/app/agents/orchestration/events.py`
- `backend/app/agents/orchestration/runner.py`
- `backend/tests/test_agent_workflow_runner.py`

已有能力：

- 顺序执行节点。
- `should_run=False` 时跳过节点。
- 节点异常时捕获失败。
- 节点输出合并到 state。
- 可选 recorder 记录运行过程。

## 3. 最小正式化内容

### 3.1 节点类型

为节点补充稳定分类，建议先使用字符串枚举语义，不急于引入复杂类层级：

- `context`：准备上下文，例如读取草稿、读取项目。
- `retrieve`：检索证据或资料。
- `generate`：调用 Agent 或 skill 生成内容。
- `validate`：校验依据、结构、合规性。
- `persist`：写入数据库。
- `render`：生成文件、HTML Deck、导出物。
- `diagnose`：生成来源诊断或质量诊断。

阶段 1 可以先在 `AgentNode` 上新增默认字段：

- `node_type = "task"`
- `label = ""`
- `visible = False`

其中 `visible` 用于区分是否适合展示给用户。当前默认 `False`，避免把执行记录重新带回产品前端。

### 3.2 节点结果规范

现有 `AgentNodeResult` 保留，但明确字段语义：

- `data_delta`：写入 workflow state 的结构化数据。
- `evidence_delta`：本节点新增的依据来源摘要。
- `messages`：可读摘要，适合进入 run 的简短消息。
- `metadata`：内部诊断元数据，可以包含 skill、耗时、数量统计。
- `error`：失败原因，不放长堆栈。

建议新增可选字段：

- `warnings: list[str]`
- `artifacts: list[dict]`

`warnings` 用于表达非阻塞风险，例如“来源限流”“证据不足”。`artifacts` 用于表达产物引用，例如 deck 文件、草稿章节、搜索任务结果。

### 3.3 事件规范

现有事件类型保留，但正式约定：

- `workflow_started`
- `node_started`
- `node_finished`
- `workflow_finished`

状态值约定：

- `pending`
- `running`
- `success`
- `failed`
- `skipped`
- `partial_success`

阶段 1 不要求业务立刻使用 `partial_success`，只把状态语义写清楚并补测试。

### 3.4 state 元数据规范

`AgentWorkflowState.metadata` 先约定以下键：

- `resolved_skills`：动作到 skill id 的映射。
- `warnings`：全局非阻塞警告。
- `artifacts`：workflow 产物引用。
- `diagnostics`：内部诊断信息。
- `recording_errors`：记录失败时的错误，不影响主流程。

- 修改：`backend/app/agents/orchestration/node.py`
- 修改：`backend/app/agents/orchestration/state.py`

## 5. 测试目标

新增或补强以下测试：

1. 节点默认包含 `node_type`、`label`、`visible`。
2. 节点返回 `warnings` 后合并到 `state.metadata["warnings"]`。
3. 节点返回 `artifacts` 后合并到 `state.metadata["artifacts"]`。
4. recorder 抛错时仍写入 `recording_errors`，主流程成功。
5. `partial_success` 状态不会被 runner 错误覆盖。

## 6. 验收标准

1. 现有 workflow 单测继续通过。
2. 首页检索、研究方向、项目设计、论文章节生成的返回结构不变。
3. 没有新增依赖。
4. 不新增前端入口。
5. 后端编排层形成清晰、可复用的契约。

## 7. 不做内容

1. 不重写现有业务 Agent。
2. 不引入 LangGraph。
3. 不新增用户可见的执行记录页面。
4. 不迁移所有 workflow 到新字段，只保证新字段向后兼容。
