# 阶段 2：执行记录与诊断标准化

## 1. 目标

把现有 `agent_workflow_runs` 和 `agent_workflow_steps` 从“基础运行记录”升级为“可诊断、可恢复、可审计”的内部记录层，同时避免把执行记录作为普通用户产品入口。

## 2. 当前基础

已有文件：

- `backend/app/models/agent_workflow.py`
- `backend/app/services/agent_workflow_record_service.py`
- `backend/app/api/agent_workflows.py`
- `backend/app/schemas/agent_workflow.py`
- `backend/tests/test_agent_workflow_records.py`

当前字段：

- run：`workflow_name`、`status`、`user_id`、`project_id`、`search_task_id`、`input_snapshot`、`output_snapshot`、`error_message`、时间字段。
- step：`run_id`、`node_name`、`status`、`input_summary`、`output_summary`、`error_message`、`duration_ms`、时间字段。

## 3. 标准化方向

### 3.1 用户摘要与内部诊断分层

run 记录建议分为两类信息：

- 用户摘要：可用于历史恢复、卡片标题、状态提示。
- 内部诊断：节点细节、skill 调用、来源状态、错误原因、耗时统计。

前端产品页只使用用户摘要，不显示内部执行记录。

### 3.2 run 建议新增字段

通过 Alembic 迁移增加：

- `workflow_version: string`
- `trigger_source: string`
- `visibility: string`
- `input_hash: string`
- `result_ref: JSONB`
- `diagnostics: JSONB`

字段语义：

- `workflow_version`：标记 workflow 版本，便于以后判断历史结果是否可复用。
- `trigger_source`：例如 `home_search`、`research_page`、`writing_page`、`ppt_page`。
- `visibility`：默认 `internal`，后续如果某些摘要可展示，再设为 `user_summary`。
- `input_hash`：用于判断同样输入是否可以复用历史结果。
- `result_ref`：保存结果引用，例如 `search_task_id`、`draft_id`、`chapter_key`、`artifact_id`。
- `diagnostics`：保存来源状态、告警、节点数量等内部摘要。

### 3.3 step 建议新增字段

通过 Alembic 迁移增加：

- `node_type: string`
- `node_label: string`
- `critical: boolean`
- `visible: boolean`
- `skill_id: string`
- `skill_version: string`
- `warnings: JSONB`
- `artifacts: JSONB`

字段语义：

- `node_type`：来自阶段 1 的节点分类。
- `node_label`：中文可读名称。
- `critical`：失败时是否中断 workflow。
- `visible`：是否适合在用户侧展示。
- `skill_id` / `skill_version`：如果节点调用 skill，记录具体能力。
- `warnings`：非阻塞问题。
- `artifacts`：本节点产物引用。

## 4. Recorder 规范

`AgentWorkflowDbRecorder` 需要统一做三件事：

1. 启动时写入 run 的基础信息、输入摘要和 `input_hash`。
2. 每个节点结束时写入节点类型、skill、warnings、artifacts、耗时。
3. workflow 结束时写入 `result_ref`、`diagnostics` 和输出摘要。

输入和输出摘要继续截断，避免把完整论文正文、大段文献摘要、隐私材料全部塞进记录。

## 5. API 边界

保留现有 `/api/agent-workflows/runs` 和 `/api/agent-workflows/runs/{run_id}`，但定位为内部诊断 API。

建议后续增加参数：

- `workflow_name`
- `project_id`
- `trigger_source`
- `visibility`

普通用户产品入口不展示 `/workflows` 页面。当前 `/workflows` 重定向到首页的策略可以保留。

## 6. 验收标准

1. Alembic 迁移可升级成功。
2. 旧记录字段兼容，不影响已有查询。
3. 新 workflow 运行后，run 能看到 `input_hash`、`result_ref`、`diagnostics`。
4. step 能看到 `node_type`、`skill_id`、`warnings`、`artifacts`。
5. 普通前端不新增执行记录按钮。

## 7. 风险控制

1. 不记录完整正文，只记录摘要和引用。
2. recorder 失败不阻塞主业务。
3. 所有新增字段先允许为空，避免历史数据迁移失败。
4. 后端 API 返回新增字段时保持前端类型向后兼容。
