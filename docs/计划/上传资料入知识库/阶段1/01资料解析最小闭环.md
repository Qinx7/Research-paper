# 阶段 1：资料解析最小闭环

## 1. 目标

先跑通“成果文件 -> 文本提取 -> 切分 -> 入库 -> 状态返回”的最小闭环。此阶段不接入学术对话，不做复杂 UI，只保证后端能力可用、可测、可回溯。

## 2. 实现边界

- 支持 TXT、MD、DOCX、PDF。
- TXT/MD 使用 UTF-8 优先，必要时兼容常见编码失败兜底。
- DOCX 使用现有 `python-docx`。
- PDF 优先检测当前环境可用解析库；若没有，实施前单独确认是否新增 `pypdf`。
- 不处理扫描件 OCR。
- 不解析图片、表格结构和公式，只提取可读正文。

## 3. 后端设计

新增 `ProjectDocumentChunk` 模型，保存上传资料的分块内容。建议字段：

- `id`：主键。
- `project_id`：所属项目。
- `outcome_id`：来源成果。
- `chunk_index`：分块序号。
- `title`：来源文件或章节标题。
- `content`：分块正文。
- `content_excerpt`：前端展示摘要。
- `source_filename`：原文件名。
- `source_type`：文件扩展名或 MIME 类型。
- `token_estimate`：粗略长度估计。
- `meta`：页码、段落范围、解析器版本等扩展信息。
- `created_at` / `updated_at`。

同时在 `Outcome.extra_data` 中维护解析状态，避免第一阶段大幅调整 `outcomes` 表结构：

```json
{
  "knowledge_status": "pending | parsing | indexed | failed",
  "knowledge_chunk_count": 12,
  "knowledge_error": null,
  "knowledge_indexed_at": "2026-06-19T00:00:00"
}
```

## 4. 服务拆分

新增 `document_parse_service.py`：

- `extract_text_from_upload(file_path_or_stream, filename) -> ParsedDocument`
- `extract_text_from_txt`
- `extract_text_from_docx`
- `extract_text_from_pdf`
- `clean_extracted_text`
- `chunk_text`

新增 `project_knowledge_service.py`：

- `index_outcome_document(db, outcome, current_user) -> ProjectDocumentIndexResult`
- `delete_outcome_chunks(db, outcome_id)`
- `update_outcome_knowledge_status(db, outcome, status, count, error)`

## 5. API 设计

在现有 `outcomes` 路由中新增最小 API：

- `POST /api/outcomes/{outcome_id}/index-knowledge`
- `GET /api/outcomes/{outcome_id}/knowledge-status`

响应示例：

```json
{
  "outcome_id": "uuid",
  "status": "indexed",
  "chunk_count": 8,
  "message": "已解析并入库",
  "error": null
}
```

## 6. 解析与切分策略

- 清洗空白、页眉页脚噪声和连续空行。
- 默认 chunk 大小按字符估算，建议 1200 到 1800 中文字符。
- chunk overlap 建议 150 到 250 字，避免跨段信息断裂。
- 单文件最大解析文本长度需要限制，避免超大文件阻塞请求。
- 第一阶段可以同步执行；如果后续大文件明显卡顿，再接 Celery 异步任务。

## 7. 测试计划

- TXT/MD：给定短文本，确认提取文本和 chunk 数正确。
- DOCX：构造小型 docx，确认段落被提取。
- PDF：使用文本型 PDF 样本，确认至少能提取正文；如果环境无 PDF 库，测试应明确跳过或断言返回“不支持 PDF 解析库”。
- 重复解析：先清理旧 chunk，再写入新 chunk，不产生重复。
- 权限：用户不能解析不属于自己的项目成果。

## 8. 验收标准

- 单个成果可通过 API 解析入库。
- 成果 `extra_data` 能记录解析状态。
- 数据库能查到对应 chunk。
- 原下载功能不受影响。
- 失败场景返回可读错误，不吞异常。
