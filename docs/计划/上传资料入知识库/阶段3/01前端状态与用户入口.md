# 阶段 3：前端状态与用户入口

## 1. 目标

在不新增复杂知识库页面的前提下，让用户能在成果管理区完成资料入库操作，并理解每个文件当前是否已经进入知识库。

## 2. 页面入口

入口放在项目详情页的成果管理区域，也就是现有 `OutcomeManager`。原因：

- 用户上传资料的位置就在成果管理里。
- 不需要新增页面和路由。
- 可以直接复用已有成果列表、下载、删除逻辑。

每条成果卡片新增解析状态：

- `未解析`
- `解析中`
- `已入库（N 段）`
- `解析失败`
- `不支持解析`

## 3. 操作按钮

根据状态显示按钮：

- 未解析：`解析入知识库`
- 已入库：`重新解析`
- 解析失败：`重试解析`
- 不支持解析：不显示主按钮，只显示原因

点击后调用：

- `POST /api/outcomes/{outcome_id}/index-knowledge`

成功后刷新成果列表，读取 `extra_data.knowledge_status` 和 `extra_data.knowledge_chunk_count`。

## 4. 文件类型提示

上传区域增加简短说明：

支持解析入知识库的文件：`TXT / MD / DOCX / 文本型 PDF`。扫描版 PDF 暂不支持 OCR。

这条说明要放在上传区，不要做成弹窗，避免干扰上传流程。

## 5. 错误展示

如果后端返回失败：

- 在成果卡片附近显示失败原因。
- 全局错误条只显示简短提示。
- 不影响文件下载和删除。

示例文案：

- `PDF 解析失败：当前文件可能是扫描件，暂不支持 OCR。`
- `文件过大：请先拆分后再解析。`
- `当前环境缺少 PDF 解析库。`

## 6. 类型与 API 封装

前端需要补充：

- `Outcome.extra_data` 中知识库状态字段的类型约定。
- `indexOutcomeKnowledge(outcomeId)`。
- `getOutcomeKnowledgeStatus(outcomeId)`，可选；第一轮也可以只依赖刷新 `listOutcomes`。

## 7. 测试计划

- 上传 TXT 后显示未解析。
- 点击解析后显示已入库和 chunk 数。
- 解析失败时显示错误信息。
- 已入库后可以重新解析。
- 前端构建通过。

## 8. 验收标准

- 用户不需要进入新页面即可把上传资料入库。
- 每个文件是否已入库清晰可见。
- 失败原因可见且不阻塞其他成果操作。
- UI 不引入新的大模块，保持项目详情页结构稳定。
