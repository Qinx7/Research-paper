# 阶段 4：稳定性验证与后续边界

## 1. 目标

在最小功能跑通后，补齐稳定性验证，明确哪些能力暂不做，避免后续误把未支持能力当成已完成。

## 2. 必做验证

后端：

- 文本解析服务单元测试。
- DOCX 解析单元测试。
- PDF 解析可用性测试。
- 成果解析 API 权限测试。
- 重复解析覆盖测试。
- 证据检索命中测试。

前端：

- `npm run build`。
- 成果管理入口人工烟测。
- 学术对话依据来源人工烟测。

## 3. 失败兜底

必须明确处理以下失败：

- 文件不存在或已被删除。
- 文件扩展名支持，但内容无法解析。
- PDF 是扫描件，提取不到文本。
- 文件过大或文本过长。
- embedding API 不可用。
- 用户无权访问成果。

## 4. 不在本轮实现

- OCR。
- PDF 原文高亮和页内定位。
- Word/PDF 批注同步。
- Elasticsearch/OpenSearch 全文检索。
- 复杂知识库管理页面。
- 多用户共享知识库权限。
- 自动解析所有历史文件。

## 5. 后续可扩展方向

- 大文件解析改成 Celery 异步任务。
- 增加项目知识库独立页面，支持查看、删除、重建 chunk。
- 支持 OCR 和扫描件 PDF。
- 支持 CSV/XLSX 结构化摘要入库。
- 将论文草稿、开题报告、项目设计也沉淀为项目知识来源。
- 增加引用级追踪：回答中的每个关键结论对应资料 chunk。
