# 阶段 2：向量化与证据检索接入

## 1. 目标

让上传资料入库后的内容真正被学术对话和论文写作召回。此阶段重点不是生成更复杂的回答，而是确保内部资料可以作为明确依据出现，并且来源可追溯。

## 2. 检索策略

先做双轨检索：

- 关键词检索作为必备能力，不依赖外部 embedding 服务。
- 向量检索作为增强能力，在 embedding 配置可用时启用。

原因是当前项目的 embedding 调用依赖外部 API。如果没有 key 或 API 限流，资料入库仍应可用，不能导致知识库完全不可检索。

## 3. 关键词召回

扩展 `evidence_retrieval_service`：

- 新增 `retrieve_project_document_chunks`。
- 使用已有 `tokenize_evidence_query` 拆词。
- 对标题、文件名、正文、元数据命中分别计分。
- 返回统一 `ProjectContextItem` 结构，`kind` 使用 `project_document_chunk`。

返回项建议字段：

```json
{
  "kind": "project_document_chunk",
  "title": "上传资料：实验记录.docx",
  "content_excerpt": "命中的资料片段...",
  "score": 8,
  "score_reasons": ["资料正文命中", "文件名命中"],
  "source_title": "实验记录.docx",
  "action_url": "/api/outcomes/{outcome_id}/download",
  "action_label": "下载来源文件"
}
```

## 4. 向量化增强

如果 embedding 可用，可以在 chunk 入库后生成向量。建议不要复用只面向论文摘要的 `document_vectors` 表，因为上传资料 chunk 和论文元数据字段不同，强行复用会造成语义混乱。

推荐做法：

- 在 `project_document_chunks` 中增加可选 `embedding` 字段，或新增 `project_document_vectors` 表。
- 第一轮更简单的做法是在 `project_document_chunks` 里保留 `embedding` 字段。
- embedding 失败不影响文本 chunk 保存。
- 检索时先关键词召回，再用向量结果补充或重排。

## 5. 对话接入

学术对话开启检索时，当前已有外部文献和项目内部证据。接入后应形成三类依据：

- 外部文献：来自 OpenAlex、CNKI、Semantic Scholar 等。
- 项目文献：用户保存到项目文献库的论文。
- 内部资料：用户上传并解析入库的文件片段。

提示词中需要明确约束：

- 上传资料片段只能作为已知材料引用。
- 不能根据资料片段编造不存在的数据、实验结果或结论。
- 如果资料不足，应明确说“当前上传资料中未找到直接依据”。

## 6. 前端依据展示

在现有聊天依据栏中增加 `project_document_chunk` 的展示文案：

- 来源类型显示“内部资料”。
- 标题显示来源文件名。
- 摘要显示命中文本片段。
- 操作按钮显示“下载来源文件”。

## 7. 测试计划

- 给项目写入几个资料 chunk，查询相关关键词能召回。
- 查询无关问题时不应强行返回 chunk。
- 学术对话返回的 `project_context_items` 包含 `kind = project_document_chunk`。
- embedding 服务不可用时，关键词召回仍通过。
- 前端类型定义能兼容新增 `kind`，构建通过。

## 8. 验收标准

- 上传资料中的明确内容能在学术对话依据来源里显示。
- 内部资料与外部文献在前端能区分。
- 无相关资料时显示无内部资料依据，不编造。
- 不影响已有项目文献和阅读笔记召回。
