# 项目资料全文搜索入口总纲

## 1. 目标

在现有“上传资料解析入知识库”能力基础上，补一个用户可直接使用的“项目资料全文搜索入口”，让用户能够主动搜索自己上传的资料内容，而不是只能被动等待学术对话或写作环节命中。

## 2. 需求理解

当前系统已经具备：

1. 成果上传
2. 资料解析入知识库
3. OCR fallback
4. 项目知识块召回
5. 写作页和项目页中的资料使用映射

但用户仍然缺少一个直接能力：

1. 我上传的资料里到底有什么内容？
2. 某个关键词在我的资料里是否出现？
3. 哪个文件、哪一段命中了当前问题？

本方案要补的是“面向用户的直接搜索入口”，不是引入 Elasticsearch / OpenSearch 级别的基础设施。

## 3. 范围边界

本轮做：

1. 基于现有 `ProjectDocumentChunk` 的项目资料搜索 API
2. 在项目页增加一个“资料全文搜索”入口区块
3. 支持关键词查询、结果列表、来源文件名、片段摘要、下载原文件

本轮不做：

1. Elasticsearch / OpenSearch
2. 高亮批注系统
3. PDF 页码级定位
4. 多条件高级检索语法
5. 单独新页面

## 4. 产品建议

建议把入口放在项目页知识工作台附近，而不是首页。

原因：

1. 这是项目私有资料能力，不适合放首页
2. 项目页已经承接成果、知识库、写作、图谱
3. 用户心智上更容易理解“在这个项目里搜我的资料”

## 5. 最小交互

用户进入项目页后：

1. 看到“项目资料搜索”输入框
2. 输入关键词
3. 返回若干命中片段
4. 每条结果显示：
   - 来源文件名
   - 片段摘要
   - 命中原因或简单相关度
   - 下载原文件按钮

## 6. 技术策略

优先复用现有数据：

1. `ProjectDocumentChunk`
2. `Outcome`
3. 现有 `_contains_any` / 关键词匹配思路

不引入新依赖，先做：

1. 关键词匹配
2. 标题 / 文件名 / 内容摘要综合排序

## 7. 阶段拆分

### 阶段1：后端搜索 API

目标：

1. 增加按项目搜索 `ProjectDocumentChunk` 的 API
2. 返回来源文件、摘要、关联 outcome 信息

### 阶段2：项目页入口

目标：

1. 在项目页增加搜索框和结果列表
2. 支持输入、清空、无结果状态

### 阶段3：结果动作闭环

目标：

1. 每条结果可下载原文件
2. 与现有知识工作台风格一致

## 8. 成功标准

1. 用户可以主动搜索项目资料内容
2. 搜索结果能指向具体来源文件
3. 不新增独立页面也能完成搜索闭环
4. 前端构建通过，后端最小测试通过
