# 证据底座补齐总纲

## 1. 需求理解

当前项目已经具备文献检索、项目文献库、文献详情、论文草稿、学术对话和部分证据展示能力，但距离需求文档中“文献驱动型科研 Agent”的闭环还差两个底座：

- 阅读笔记与证据卡片：需要把单篇文献分析、用户摘录、关键结论、可引用依据沉淀为可维护数据，而不是只在页面上临时展示。
- 检索任务化：需要把一次检索升级为可追踪、可复用、可诊断的检索任务，记录模式、范围、来源状态、结果摘要和失败原因。

本方案只补齐现有功能闭环，不扩展新的大型 Agent，不新增复杂依赖，不改变已有页面主路径。

## 2. 关键假设

- 继续使用现有 FastAPI、SQLAlchemy、PostgreSQL/JSONB、Next.js 架构。
- 数据库迁移沿用项目当前建表/初始化方式；如果后续确定使用 Alembic，再单独补迁移。
- 阅读笔记先绑定项目文献 `Paper`，不先支持任意外部 URL 或 PDF 全文批注。
- 检索任务先服务首页检索和学术对话，不强制替换所有旧接口；旧接口保持兼容。
- 学术对话只要有相关依据来源即可，不要求所有回答都强制生成长篇引用体系。

## 3. 总体目标

补齐后，项目应具备以下能力：

- 用户可以在文献详情页保存阅读笔记、关键观点、证据摘录和可靠性标记。
- 系统可以把单篇文献结构化分析结果沉淀为证据卡片，并在后续论文写作和学术对话中复用。
- 每次学术检索都能形成任务记录，保存查询参数、来源状态、返回结果和错误摘要。
- 当 CNKI、Semantic Scholar、OpenAlex 等来源限流或失败时，前端可以展示任务级诊断，而不是只表现为空结果。
- 学术对话的依据展示可以同时显示外部文献和项目内部证据卡片。

## 4. 分阶段实施

### 阶段 1：阅读笔记与证据卡片最小闭环

先新增 `paper_notes` 数据模型和最小 API，让文献详情页可以保存、读取、更新、删除笔记。第一步只做用户可见闭环，不做复杂全文批注。

### 阶段 2：检索任务化最小闭环

新增 `literature_search_tasks` 数据模型和任务 API。先让后端在检索时保存任务记录，再让首页和学术对话读取任务状态。

### 阶段 3：对话与写作复用证据底座

把阅读笔记、证据卡片和检索任务结果接入学术对话、论文草稿引用建议和文献矩阵。此阶段只做复用，不重写已有 Agent。

### 阶段 4：证据卡片检索排序增强

把阶段 3 中分散在聊天和写作里的轻量匹配逻辑抽成统一服务。先做可诊断、可测试的关键词评分，再逐步补充来源权重、证据质量权重和前端诊断展示，避免内部证据“有但召回不到”或“不相关也被塞进上下文”。

## 6. 验收标准

- 文献详情页能新增、编辑、删除阅读笔记。
- 单篇文献分析结果可以生成至少一条可展示的证据卡片。
- 首页学术检索完成后，后端能保存一条检索任务记录，包含查询、模式、范围、来源状态和结果数量。
- 开启学术检索的对话中，如果存在项目内部笔记或证据卡片，回答下方能看到内部依据。
- 当来源失败或限流时，任务记录中能看到具体来源状态，前端展示“暂无相关文献”或失败原因，而不是空白。
- 后端相关单元测试通过，前端构建通过。

## 7. 需要确认的事项

- 是否允许新增两个数据库表：`paper_notes`、`literature_search_tasks`。
- 阅读笔记是否先只做项目内文献，不做 PDF 原文划线批注。
- 检索任务历史是否需要在首页展示独立列表，还是先只在当前结果页和学术对话中使用。
