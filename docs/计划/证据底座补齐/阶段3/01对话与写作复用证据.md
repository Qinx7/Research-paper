# 阶段 3 步骤 01：对话与写作复用证据

## 目标

让学术对话和论文写作能复用 `paper_notes` 中的阅读笔记与证据卡片，减少回答和生成材料中的空泛内容。

## 后端接入

修改 `backend/app/api/chat.py` 中的项目私有上下文构建逻辑：

- 当前已整合项目、设计、开题、草稿、成果。
- 增加读取当前项目下最近或最相关的 `PaperNote`。
- 按问题关键词做轻量匹配：标题、内容、标签、证据摘录。
- 输出到 `project_context_items`，类型标记为 `paper_note`。

建议上下文格式：

```text
内部证据卡片：
1. 标题：...
   类型：finding
   来源文献：...
   证据摘录：...
   可靠性：...
```

## 前端接入

修改 `frontend/src/components/chat/ChatMessages.tsx`：

- `ProjectEvidenceCard` 支持 `paper_note` 类型。
- 卡片展示笔记标题、类型、证据摘录、来源文献。
- 如果同时存在外部文献和内部笔记，分组展示。

## 写作接入

修改 `backend/app/api/drafts.py` 的文献上下文构建：

- 在 `_build_literature_context` 中补充高置信度 `PaperNote`。
- 章节生成 prompt 中把证据卡片作为“可引用依据”，要求只基于这些依据表达。

## 测试计划

后端：

```powershell
cd backend
.\.venv\Scripts\python.exe -m unittest tests.test_chat_search_threading tests.test_grounding_guard tests.test_paper_notes_api
```

前端：

```powershell
cd frontend
npm.cmd run build
```

人工验证：

- 在一篇文献详情页保存证据卡片。
- 打开学术对话，选择对应项目并提问相关问题。
- 回答下方出现内部依据卡片。
- 生成论文某章节时，内容能引用项目内证据，不凭空扩展不存在的数据。

## 风险

- 轻量关键词匹配可能召回不准，但足够作为第一步验证。
- 后续如果要更强检索能力，可以再引入向量化；本步骤不新增依赖。
