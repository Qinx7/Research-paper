# 未完成功能补齐总纲

## 1. 需求理解

当前目标不是继续零散修补，而是对照 `README.md`、`literature_driven_research_agent_prd.md` 和现有代码，把“文案已承诺、方案已规划、但产品闭环仍未完成”的功能补齐，并按优先级推进。

本轮总纲重点关注三类缺口：

1. 已有后端能力，但前端入口、状态闭环或数据沉淀不完整。
2. 已有页面和接口，但一致性较差，导致能力名义存在、实际不好用。
3. PRD 中明确承诺，但当前代码里仍缺少关键基础设施或最小产品化入口。

## 2. 当前缺口重新归类

### 2.1 优先级最高

1. 项目私有知识库闭环不完整  
   上传成果、文献笔记、草稿章节、对话证据、项目设计还没有统一进入同一套可检索知识库。

2. 论文写作工作台存在双入口分叉  
   `/writing` 和 `PaperWorkflow` 都能写论文，但状态、引用、章节规则仍有重复实现和漂移风险。

3. 开题主流程缺少稳定连续入口  
   需求理解、文献分析、研究方向、项目设计、开题报告、开题 PPT 后端能力分散存在，但没有明确的一条主流程入口。

4. 学术搜索质量筛选仍停留在“部分能力可用”  
   已接入多来源和部分权威标签，但权威目录导入、细粒度筛选和来源诊断还不完整。

### 2.2 中优先级

1. 文献全文解析与证据沉淀深度不足  
   OCR、解析、切分已存在，但对不同来源 PDF/扫描件/成果文档的统一沉淀还不稳定。

2. 答辩与交付链路未产品化  
   答辩 PPT、PDF 导出、演讲稿、问题预测能力分散存在，但统一交付工作台不足。

3. 异步任务体验薄弱  
   任务执行有记录，但失败重试、阶段性进度、历史追踪仍偏基础。

### 2.3 低优先级但仍属 PRD 缺口

1. 导师审阅与团队协作
2. 全文检索引擎（Elasticsearch / OpenSearch）
3. 数据库迁移体系（Alembic）
4. 成果包归档与权限角色扩展

## 3. 建议实施顺序

### 阶段1：知识库与写作闭环

目标：先把“搜索到内容、沉淀为证据、进入写作、在写作中可追溯引用”这一条链路做稳。

包含：

1. 统一草稿章节引用来源和证据来源
2. 把草稿章节、阅读笔记、项目设计、成果文档统一纳入项目知识库检索
3. 收敛 `/writing` 与 `PaperWorkflow` 的状态规则

### 阶段2：开题主流程产品化

目标：把现有分散能力组织成一条明确入口，而不是让用户自己跳页面拼流程。

包含：

1. 需求理解 → 文献分析 → 研究方向 的稳定连续入口
2. 研究方向 → 项目设计 → 开题报告 / 开题 PPT 的承接入口
3. 阶段性结果沉淀到项目

### 阶段3：学术搜索质量增强

目标：把现在“能搜到”提升为“能解释来源、能筛质量、能说明为什么选这些文献”。

包含：

1. PubScholar / CNKI / 维普 / PubMed / OpenAlex 等来源诊断完善
2. 权威目录导入接口预留
3. 来源聚合、命中解释、搜索精度调优

### 阶段4：交付与答辩链路

目标：把论文、报告、PPT、PDF、演讲稿整成一个交付工作台。

包含：

1. 统一导出入口
2. 答辩材料生成与预览
3. 任务历史和交付记录

### 阶段5：协作与基础设施

目标：补齐 PRD 中真正的大项，但不抢前面主链路资源。

包含：

1. 导师/团队协作
2. Alembic 迁移
3. 全文检索引擎
4. 权限与归档

## 4. 建议本轮先做什么

建议从 **阶段1** 开始，而不是继续扩散到更多页面。

原因：

1. 现有功能里最直接影响“能不能稳定使用”的，已经变成知识沉淀和写作一致性。
2. 如果阶段1不先收敛，后面开题主流程和交付链路会继续建立在分叉状态之上。
3. 阶段1的成果能直接提升搜索、对话、写作三个核心场景。

## 5. 风险与边界

1. 不建议“一次性完成全部未实现功能”，会显著扩大回归面。
2. `/writing` 与 `PaperWorkflow` 如果不先收敛状态模型，后续任何新功能都会出现双份实现。
3. PubScholar 当前已接入浏览器会话模式，但搜索精度还需要单独调优，不应与更大范围功能补齐混在一起。

## 6. 需要确认

建议你确认以下执行起点：

1. 是否按 **阶段1：知识库与写作闭环** 先做
2. 阶段1内部是否接受我先从“统一项目知识库入口 + 写作状态收敛”开始
