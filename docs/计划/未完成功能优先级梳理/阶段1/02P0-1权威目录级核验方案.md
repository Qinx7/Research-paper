# P0-1 权威目录级核验方案

## 1. 目标

把当前学术搜索里的权威标签体系从“规则识别 + 待核验提示”为主，补到“可运行的目录级核验能力 + 前后端统一语义展示”。

本轮目标不是覆盖所有外部商业目录，而是先把当前已经对外承诺的能力做成最小可信闭环：

1. 系统能识别哪些标签是“目录级已核验”
2. 系统能识别哪些标签只是“待核验”
3. 前端能稳定解释这两类状态
4. 本地环境不配置目录时，行为也一致可解释

## 2. 当前现状

### 已有能力

- 后端已有 `authority_filter_service.py`
- 后端已有 `authority_catalog_service.py`
- 检索结果里已经产出：
  - `authority_tags`
  - `pending_authority_tags`
  - `authority_reasons`
  - `authority_level`
  - `verified_level`
- 前端首页与解释层已经消费这些字段

### 当前缺口

1. 目录级核验默认不可用
   - 当前依赖 `AUTHORITY_CATALOG_PATH`
   - 但没有内建样例目录、目录格式说明、接入校验和产品化说明

2. 状态语义还不够硬
   - `IEEE / ACM` 现在既有规则命中，也可能表现为“已核验”
   - `EI / JCR / 中科院分区` 更多是“待核验”
   - 但这些差异没有被前端统一解释成用户可理解的层级

3. 结果解释层还不够统一
   - 首页卡片、解释层、搜索总览、学术对话来源展示还没有完全统一“已核验 / 待核验 / 来源标签”三种概念

## 3. 产品边界

### 本轮要做

1. 明确权威标签状态模型
2. 提供本地目录样例接入能力
3. 统一前后端展示与解释文案
4. 为首页搜索结果和对话来源层提供一致说明

### 本轮不做

1. 在线抓取商业目录
2. 自动同步第三方付费目录
3. 完整目录运营后台
4. 实时订阅外部目录更新

## 4. 建议状态模型

建议把当前标签状态收敛为三层：

### A. 已核验

含义：

- 来自本地授权目录命中
- 或来自规则即可强确定的来源级标签

适用：

- `IEEE`
- `ACM`
- `北大核心`
- 配置目录后命中的 `EI / JCR / 中科院分区`

### B. 待核验

含义：

- 文本、来源或元数据中出现相关信号
- 但尚未命中本地授权目录

适用：

- `EI`
- `JCR`
- `中科院分区`

### C. 来源标签

含义：

- 只是数据源或出版方信息
- 不等同于目录级权威认证

适用：

- `OpenAlex`
- `Semantic Scholar`
- `PubScholar`
- `CNKI`
- `VIP`

## 5. 数据接入建议

### 5.1 最小目录格式

建议优先支持：

1. `CSV`
2. `JSON`

字段最小集合：

- `title`
- `doi`
- `venue`
- `tags`

`tags` 支持：

- `ei`
- `jcr`
- `cas`
- `pku_core`
- `ieee`
- `acm`

### 5.2 样例目录

建议新增一份可直接跑通的样例目录，例如：

- `backend/storage/authority_catalogs/sample_authority_catalog.csv`

用途：

1. 让本地开发环境不再完全空转
2. 让 `AUTHORITY_CATALOG_PATH` 有明确参考
3. 让测试可以基于固定样例验证

### 5.3 配置说明

建议在：

- `backend/.env.example`
- `README.md`

补充：

1. `AUTHORITY_CATALOG_PATH` 的格式说明
2. 样例目录路径
3. 未配置目录时系统行为说明

## 6. 前端落点

### 首页搜索结果卡片

位置：

- `frontend/src/components/HomeSearchLanding.tsx`
- `frontend/src/lib/authorityBadges.mjs`

要点：

1. 已核验标签和待核验标签视觉上明确区分
2. Tooltip 或解释文案说明：
   - 已核验：已命中本地授权目录或强确定来源规则
   - 待核验：仅检测到相关信号，未完成目录级确认

### 搜索解释层

位置：

- `frontend/src/lib/searchExplain.mjs`

要点：

1. 统一产出“为什么是已核验 / 为什么是待核验”
2. 不再把待核验措辞写成像已确认一样

### 学术对话来源层

位置：

- 聊天来源卡片相关组件和类型

要点：

1. 对话中的来源展示复用同一套 authority badge 逻辑
2. 避免首页和对话页对同一篇论文显示不同语义

## 7. 后端落点

### 核心服务

- `backend/app/services/authority_filter_service.py`
- `backend/app/services/authority_catalog_service.py`

建议补强点：

1. 明确哪些标签允许规则直接进入“已核验”
2. 明确哪些标签只能目录命中后进入“已核验”
3. 把 `reason` 组织成更稳定的分类文案

### 搜索主链

- `backend/app/agents/literature_search_agent.py`

建议补强点：

1. 统一注入 `authority_tags / pending_authority_tags`
2. 搜索总览里增加目录核验统计说明

## 8. 推荐实施顺序

### 阶段1：最小可用

1. 新增样例目录
2. 明确 `AUTHORITY_CATALOG_PATH` 配置方式
3. 补后端测试：
   - 目录命中
   - 未命中时转为待核验
   - `IEEE / ACM` 强规则命中

### 阶段2：前端统一

1. 首页卡片统一 badge 语义
2. 搜索解释层统一文案
3. 对话来源层复用同一逻辑

### 阶段3：收口

1. README 与 `.env.example` 补说明
2. 搜索总览补统计说明
3. 样例目录验证链补完

## 9. 需要确认的事项

### 需要你确认

1. `EI / JCR / 中科院分区` 本轮是否接受“本地样例目录 + 用户自备正式目录文件”的落地方式
2. `IEEE / ACM` 是否继续视为“规则可强确定的已核验标签”
3. 是否要把“未配置目录时”的首页提示做得更显式

## 10. 推荐结论

如果按最小改动落地，推荐采用：

1. `IEEE / ACM` 保持规则直判已核验
2. `EI / JCR / 中科院分区` 必须目录命中才算已核验
3. 未命中目录时统一显示“待核验”
4. 先提供样例目录和接入说明，不尝试在线抓外部目录
