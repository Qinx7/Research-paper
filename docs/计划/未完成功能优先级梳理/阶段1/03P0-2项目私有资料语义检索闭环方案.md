# P0-2 项目私有资料语义检索闭环方案

## 1. 目标

把当前“上传资料进入知识库”的能力，从“可切块入库 + 关键词搜索”升级为“项目私有资料可向量化、可语义召回、可在写作和对话中复用”的闭环能力。

本轮目标不是做泛化全文检索平台，而是先完成当前主链路真正依赖的最小闭环：

1. 上传成果文件后，`ProjectDocumentChunk` 不只是文本入库，还能进入项目私有向量检索
2. 项目页资料搜索同时支持关键词召回和语义召回
3. 写作页与学术对话能稳定拿到项目内资料依据，而不只是文献和证据卡片

## 2. 当前现状

### 已有能力

1. 成果文件上传
2. PDF / 文本资料解析
3. `ProjectDocumentChunk` 切块入库
4. 项目页 `Document Search`
5. 写作 / 对话里已有项目内证据召回骨架

### 当前缺口

1. `ProjectDocumentChunk` 没有进入独立的项目私有向量检索
2. 项目资料搜索仍是关键词/规则打分
3. 证据召回服务对资料片段仍是关键词召回，不是语义召回
4. 项目私有资料与论文文献向量库是分裂的两条链

## 3. 产品边界

### 本轮要做

1. 为项目资料 chunk 增加向量化存储
2. 为项目资料增加语义检索 API 路径
3. 为证据召回服务增加“关键词 + 语义”混合召回
4. 让项目页、写作页、对话页都能消费这一能力

### 本轮不做

1. Elasticsearch / OpenSearch
2. 跨项目全局资料搜索
3. 复杂 rerank 服务
4. 多向量索引实验

## 4. 建议架构

### 4.1 数据层

当前已有：

- `document_vectors`

建议新增：

- `project_document_vectors`

最小字段建议：

1. `chunk_id`
2. `project_id`
3. `outcome_id`
4. `title`
5. `source_filename`
6. `source_type`
7. `section_title`
8. `content_excerpt`
9. `embedding`

原因：

- 不和论文文献向量表混用
- 检索时天然限定在项目内
- 结果可直接回跳到项目资料片段

### 4.2 服务层

建议新增一条独立服务：

- `backend/app/services/project_document_embedding_service.py`

职责：

1. 为 `ProjectDocumentChunk` 生成 embedding
2. 维护 `project_document_vectors`
3. 执行项目内语义检索

### 4.3 检索层

当前：

- `project_document_search_service.py` 只做关键词打分

建议改成混合召回：

1. 关键词召回
2. 语义召回
3. 去重合并
4. 输出统一结果结构

## 5. 页面落点

### 项目页

位置：

- `frontend/src/app/projects/[id]/page.tsx`

本轮目标：

1. 资料搜索命中结果里可解释“关键词 / 语义”命中原因
2. 空结果时区分：
   - 未入知识库
   - 已入库但未命中
   - 向量未就绪

### 写作页

位置：

- `/writing`

本轮目标：

1. 写作依据里能更稳定拿到项目资料片段
2. 后续整篇生成 / 审查 / 修订可以复用项目资料语义召回

### 首页学术对话

位置：

- `HomeSearchLanding`
- 聊天来源展示层

本轮目标：

1. 开启学术检索时，项目资料依据不只靠关键词命中
2. 来源层能区分：
   - 外部文献依据
   - 项目资料依据

## 6. 推荐实施顺序

### 阶段1：数据闭环最小实现

1. 新增 `project_document_vectors`
2. 为 `ProjectDocumentChunk` 建立向量化服务
3. 在成果入库后同步生成 chunk embedding

### 阶段2：检索闭环

1. 扩展 `project_document_search_service.py`
2. 增加项目资料语义检索函数
3. 做关键词 + 语义混合结果合并

### 阶段3：证据复用闭环

1. `evidence_retrieval_service.py` 接入项目资料语义召回
2. 写作页与对话页统一消费项目资料依据

## 7. 关键文件建议

### 后端

- 修改：
  - `backend/app/services/project_knowledge_service.py`
  - `backend/app/services/project_document_search_service.py`
  - `backend/app/services/evidence_retrieval_service.py`
  - `backend/app/services/embedding_service.py`
  - `backend/app/api/projects.py`

- 新增：
  - `backend/app/services/project_document_embedding_service.py`
  - `backend/app/models/project_document_vector.py` 或沿用 SQL 直表
  - 对应测试文件

### 前端

- 修改：
  - `frontend/src/app/projects/[id]/page.tsx`
  - `frontend/src/lib/api.ts`
  - `frontend/src/lib/types.ts`
  - 资料搜索解释相关 `mjs` 工具

## 8. 风险点

1. 当前环境 embedding 调用依赖外部模型服务
2. 项目资料 chunk 数量大时，入库后同步向量化可能变慢
3. 需要避免项目资料向量表和论文文献向量表混淆

## 9. 推荐结论

推荐按下面的实现边界执行：

1. 项目资料单独建向量表
2. 上传成果入库后同步生成 chunk 向量
3. 项目页资料搜索改成关键词 + 语义混合召回
4. 写作页和对话页统一复用这条项目资料依据链

## 10. 需要确认的事项

1. 本轮是否接受“项目资料向量化采用同步生成”，而不是异步任务化
2. `project_document_vectors` 是否接受独立表方案，而不是继续复用 `document_vectors`
