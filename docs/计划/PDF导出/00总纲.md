# PDF 导出方案总纲

## 1. 需求理解

当前项目支持论文草稿和开题报告的 DOCX 导出（python-docx），但缺少 PDF 格式。学术场景中 PDF 更通用、排版更稳定、不易被篡改，是提交论文和开题报告的刚需格式。

本次目标：在现有 DOCX 导出基础上增加 PDF 导出能力，不影响现有流程。

## 2. 关键设计决策

### 决策 1：fpdf2 直接生成 PDF（零系统依赖）

选用 `fpdf2`（纯 Python PDF 库），与现有 `_build_docx()` 模式平行，新增 `_build_pdf()` 函数。

- 纯 Python 实现，Windows/Linux 均可直接运行
- 无需安装 LibreOffice、wkhtmltopdf 等系统级工具
- API 清晰，中文支持通过嵌入 TTF 字体实现
- 新依赖仅一行：`fpdf2`（约 2MB）

### 决策 2：抽取通用 PDF 构建器模块

论文草稿和开题报告两处的 PDF 导出逻辑高度相似（A4 纸、页码、章节标题格式），抽取为独立工具模块避免重复代码：

- 新建 `backend/app/services/pdf_builder.py`
- 提供 `PdfBuilder` 类：封装纸张设置、字体注册、标题/正文/页码渲染
- `_build_pdf(draft/proposal)` 只需传入结构化内容和标题

### 决策 3：中文字体自动检测 + 捆绑回退

fpdf2 需要 TTF 字体文件来渲染中文：

1. 优先检测系统字体（Windows：`C:\Windows\Fonts\simsun.ttc`，Linux：`/usr/share/fonts/`）
2. 若未找到 → 自动下载开源字体（Noto Sans SC，约 10MB）到 `storage/fonts/`
3. 首次运行时一次性完成，后续直接读取

不使用 SimSun 作为捆绑字体（版权问题），使用 SIL Open Font License 的 Noto Sans SC。

### 决策 4：不改造现有 DOCX 端点，新增 `?format=pdf` 查询参数

现有 `/download` 端点默认返回 DOCX，新增 `?format=pdf` 参数切换输出格式。前端在下载按钮旁增加格式选择下拉框。

## 3. 实施阶段

### 阶段 1：基础设施（PDF 构建器 + fpdf2 + 字体）

- 安装 `fpdf2`
- 新建 `backend/app/services/pdf_builder.py`（PdfBuilder 类 + 字体检测）
- 验证中文渲染正确

### 阶段 2：后端 PDF 导出端点

- `drafts.py`：`GET /api/drafts/{id}/download?format=pdf`
- `proposal.py`：`GET /api/proposal/{id}/download?format=pdf`
- 与现有 DOCX 导出共用存储/缓存逻辑

### 阶段 3：前端下载入口

- PaperWorkflow.tsx：论文草稿下载增加格式选择（DOCX / PDF）
- PipelineWizard.tsx 第 6 步：开题报告下载增加格式选择

## 5. 验证方式

1. `pip install fpdf2` → 安装成功
2. 启动后端 → 首次请求 PDF 时自动下载字体到 `storage/fonts/`
3. `curl -X GET "http://localhost:8000/api/drafts/{id}/download?format=pdf" -o test.pdf` → 文件可正常打开，中文无乱码
4. `curl -X GET "http://localhost:8000/api/proposal/{id}/download?format=pdf" -o test.pdf` → 同上
5. 前端 → 论文阶段/开题阶段 → 选择 PDF 格式下载 → 文件正确
