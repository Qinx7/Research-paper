# 研究页连续承接重构总纲

## 1. 目标

把当前 `/research` 从“研究方向分析页”重构为更产品化的“研究方案详情页”，保留双栏结构，但弱化流程感和工作流感，避免回到此前 `pipeline` 页面那种形态。

## 2. 需求理解

当前研究页存在的问题：

1. 页面重点仍偏向“分析展示”，而不是“研究方案承接”
2. 虽然已经串了研究方向、项目设计、开题材料，但用户感知更像在看多个分析模块
3. 页面信息密度高，热度、趋势、评分等内容过强，压过了真正要继续使用的研究方案主体
4. 若进一步加强步骤条、流程推进，会重新接近用户明确不需要的 `pipeline / 工作流` 风格

本次重构的产品目标：

1. 保留左侧候选方向切换能力
2. 右侧改成完整的研究方案详情区，不做流程编排界面
3. 默认承接顺序为：
   研究方向详情 → 项目设计 → 开题材料
4. 开题报告与开题 PPT 统一归并到“开题材料”区
5. 方向切换先预览，不自动设为当前方向
6. 用户通过明确按钮确认“设为当前研究方向”

## 3. 已确认产品决策

### 3.1 页面结构

采用“双栏增强型”：

1. 左侧：候选研究方向列表
2. 右侧：当前方向的研究方案详情页

### 3.2 右侧内容顺序

1. 研究方向详情
2. 项目设计
3. 开题材料（统一包含开题报告、开题 PPT）

### 3.3 方向切换策略

1. 左侧点击方向只切换预览
2. 不自动写回当前项目主方向
3. 由右侧“设为当前研究方向”按钮显式确认

### 3.4 信息收敛策略

1. 保留关键评分
2. 保留研究空白
3. 弱化或移除趋势图、过多图表化展示
4. 页面重点放在“当前研究方案内容”和“后续承接动作”

## 4. 非目标

本轮不做：

1. 不恢复 `pipeline` 式工作流页面
2. 不新增新的后端数据模型
3. 不扩展研究页之外的新主链路
4. 不把研究页改造成全屏多步骤向导
5. 不同步重构首页、项目页、写作页的信息架构

## 5. 实施原则

1. 先重构现有页面结构和状态承接，不改业务能力边界
2. 尽量复用现有 `research direction / design / proposal / ppt` 接口
3. 优先最小可用改动，不引入新依赖
4. 先保证预览态、确认态、已承接态清晰，再处理视觉细节

## 6. 阶段拆分

### 阶段1：研究页结构收敛

目标：

1. 右侧从“分析面板”收敛为“研究方案详情页”
2. 调整信息层级
3. 弱化图表展示

### 阶段2：当前方向确认机制

目标：

1. 区分“预览方向”与“当前方向”
2. 提供明确的“设为当前研究方向”动作
3. 让后续设计与开题材料建立在“当前方向”之上

### 阶段3：项目设计与开题材料整合

目标：

1. 项目设计区更像承接区，而不是单个按钮区域
2. 开题报告与开题 PPT 合并为统一“开题材料”区
3. 明确未生成 / 已生成 / 可下载三种状态

### 阶段4：验证与回归

目标：

1. 保证 `/research` 不回退为工作流页感受
2. 验证从首页、项目页进入 `/research` 的承接仍然成立
3. 验证现有生成动作不回归

## 7. 风险

1. 当前 `frontend/src/app/research/page.tsx` 文件已较长，直接堆功能会继续恶化可维护性
2. “预览方向”和“当前方向”如果状态设计不清，会出现按钮文案和结果归属混乱
3. 若一次性重做太多视觉层次，容易影响现有生成/下载动作稳定性

## 8. 成功标准

1. 用户进入 `/research` 时，感知是“研究方案详情页”，不是流程编排台
2. 左侧可切换候选方向，右侧能稳定预览不同方向
3. 用户可明确确认当前研究方向
4. 项目设计与开题材料都围绕当前方向承接
5. 页面中不再由大块趋势图表主导视觉注意力
