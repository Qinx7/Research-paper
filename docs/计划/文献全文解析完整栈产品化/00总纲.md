# 文献全文解析完整栈产品化总纲

## 1. 目标

在现有“上传资料解析入知识库”能力基础上，把全文解析从“可用的最小闭环”升级成“更完整、可持续扩展的解析栈”，重点补齐复杂 PDF、版面差异和文献类材料解析能力，而不是立即引入重型全文搜索平台。

## 2. 现状判断

当前项目已经具备：

1. TXT / MD / DOCX / PDF 解析
2. `pypdf` 文本型 PDF 提取
3. `PyMuPDF + pytesseract + Tesseract` OCR fallback
4. 解析后切分为 `ProjectDocumentChunk`
5. 项目资料搜索、写作引用、章节映射

这说明“上传资料入知识库”已经不是空白。

但与原始需求文档相比，仍有明显差距：

1. 没有 `GROBID` 学术论文结构化解析
2. 没有 `pdfplumber` 版面补充解析
3. 没有 `PaddleOCR` 路线
4. 没有按文献类型区分解析策略
5. 没有更强的章节、参考文献、页码、结构字段提取

## 3. 范围边界

本轮做：

1. 梳理并升级 PDF / 文献类材料解析链路
2. 为后续接入 `GROBID / pdfplumber / PaddleOCR` 留出明确结构
3. 优先增强“论文型 PDF / 扫描型 PDF / 普通资料 PDF”的解析稳定性

本轮不做：

1. Elasticsearch / OpenSearch
2. 文献页码高亮批注系统
3. 完整参考文献自动纠错平台
4. 单独独立全文阅读器

## 4. 推荐策略

### 4.1 解析链分层

把 PDF / 文献解析明确拆成三层：

1. 纯文本优先层  
   - `pypdf`
   - 适合文本型 PDF

2. 版面补充层  
   - `pdfplumber`
   - 适合布局复杂、`pypdf` 提取差的 PDF

3. OCR fallback 层  
   - 当前已有：`PyMuPDF + pytesseract`
   - 未来可选：`PaddleOCR`

### 4.2 文献型材料增强

对于“学术论文 PDF”：

1. 未来接入 `GROBID`
2. 尝试抽取：
   - 标题
   - 摘要
   - 章节结构
   - 参考文献区

但本轮先以接入位和解析结构设计为主，不承诺一次性做完全部学术结构抽取。

## 5. 阶段拆分

### 阶段1：现有解析链重构为多策略结构

目标：

1. 明确解析策略顺序
2. 将 PDF 解析从单一路径重构为可扩展策略链

### 阶段2：补 `pdfplumber` 版面解析

目标：

1. 对 `pypdf` 提取效果差但又不是扫描件的 PDF，提供第二层补充提取

### 阶段3：论文型 PDF 结构化预留

目标：

1. 为 `GROBID` 接入预留结构
2. 明确论文类文档和普通资料文档的解析差异

### 阶段4：OCR 路径产品化补强

目标：

1. 让 OCR 失败原因更可解释
2. 让扫描件解析状态更明确
3. 为未来 `PaddleOCR` 替换或并行接入留接口

## 6. 成功标准

1. 文本型 PDF、复杂版面 PDF、扫描型 PDF 有更清晰的解析策略
2. 当前解析服务具备可扩展结构，而不是继续堆单函数逻辑
3. 用户在资料入库失败时能看到更明确的原因
4. 后续若要接 `GROBID / pdfplumber / PaddleOCR`，不需要重写整条链路
