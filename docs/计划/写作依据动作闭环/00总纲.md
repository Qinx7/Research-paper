# 写作依据动作闭环总纲

## 1. 需求理解

当前写作页已经能够展示：

- 当前章节命中的成果材料
- 当前章节命中的资料片段
- 当前章节关联的项目文献
- 当前章节关联的证据卡片

项目页也已经能够看到部分“哪些成果被章节引用”的信息。

但从用户角度看，当前仍然是“看得到依据”，还不是“依据动作闭环”：

1. 每条依据的跳转目标不完全统一；
2. 用户不容易判断“点进去后会看到什么”；
3. 写作页里还缺少更明确的“确认本章引用来源”动作；
4. 项目页虽然能看章节使用情况，但反查链路仍不够完整。

本方案的目标不是再增加更多依据类型，而是把现有依据链路收口成用户可理解、可操作、可回查的一套闭环。

## 2. 关键假设

- 优先复用现有 `project workspace` 返回结构，不优先改后端模型。
- 优先做前端动作统一、来源文案统一、跳转路径统一。
- 不引入复杂“引用管理系统”或富文本标注协议。
- 先解决“打开依据后用户知道自己到了哪里”这个问题。

## 3. 当前已具备的基础

### 3.1 写作页

已有：

1. `chapterKnowledgeActions.outcomes`
2. `chapterKnowledgeActions.chunks`
3. `chapterKnowledgeActions.papers`
4. `chapterKnowledgeActions.notes`

说明：
这些动作已经能生成 `href / actionLabel / external`，说明后端与前端的数据底座已经有了。

### 3.2 项目页

已有：

1. 项目知识工作台
2. 项目资料全文搜索
3. 项目文献库
4. 章节使用情况的部分展示

说明：
当前差的不是完全没有入口，而是“动作和动作之间还没有完全闭环”。

## 4. 核心目标

增强后，用户在写作场景中应能明确完成这 4 件事：

1. 知道本章用了哪些依据；
2. 知道每条依据点开后会去哪里；
3. 能从依据快速回到最准确的项目页面位置；
4. 能在项目页反查“哪些章节正在用这条依据”。

## 5. 分阶段实施

### 阶段 1：写作页来源动作统一

目标：

- 统一成果、资料片段、文献、证据卡片在写作页中的动作语义。

最小实现：

1. 为每类依据统一动作文案：
   - 成果：`查看成果`
   - 资料片段：`查看片段`
   - 文献：`查看文献`
   - 证据：`查看证据`
2. 在写作页中补一层更明确的来源说明文字；
3. 避免同样是跳转，却一会儿叫“查看来源”、一会儿叫“下载资料”。

### 阶段 2：写作到项目页的跳转定位增强

目标：

- 用户从写作页点开依据后，能尽量落到更准确的位置，而不是只跳到一个泛页面。

最小实现：

1. 统一项目页的 `view` / `tab` / `highlight` 参数语义；
2. 依据动作尽量带：
   - 目标模块
   - 对象 id
   - 可选高亮标识
3. 项目页接收这些参数后自动展开或高亮对应内容。

### 阶段 3：项目页反查章节引用增强

目标：

- 在项目页中更明确展示“这条成果 / 资料 / 文献正被哪些章节使用”。

最小实现：

1. 强化章节引用列表展示；
2. 为每个章节提供“跳到写作页”的动作；
3. 保证从项目页回写作页时能直接定位到对应章节。

## 6. 不做事项

本轮不做：

1. 富文本级逐段引用锚点
2. 正式参考文献管理器
3. 自动插入脚注
4. 复杂引用确认审核流

## 7. 建议修改文件

- `frontend/src/app/writing/page.tsx`
- `frontend/src/app/projects/[id]/page.tsx`
- `frontend/src/lib/projectKnowledge.mjs`
- 如确有必要再补：
  - `frontend/src/lib/types.ts`
  - `backend/app/services/project_workspace_service.py`

## 8. 验收标准

- 写作页的依据动作文案统一；
- 用户能从写作页顺手打开最准确的来源位置；
- 项目页能更明确显示“哪些章节使用了这条依据”；
- 用户能从项目页跳回对应章节；
- 不新增复杂引用协议，不破坏当前写作主链路。
