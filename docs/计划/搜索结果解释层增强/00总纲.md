# 搜索结果解释层增强总纲

## 1. 需求理解

当前首页学术检索已经能返回：

- 文献结果列表
- 权威标签
- 来源状态诊断
- 部分 `why_selected`
- 部分 `authority_reasons`

但这些解释信息还比较分散，用户很难快速回答下面几个问题：

1. 这篇为什么会排在前面？
2. 这篇是因为相关性高，还是因为来源权威？
3. 这个标签是官方核验，还是系统推断？
4. 某个来源没返回，是因为没文献，还是因为限流/超时/被拦截？
5. 当前这次搜索整体上“可信到什么程度”？

本方案目标不是增加更多检索源，而是把现有结果解释层做清楚、做统一、做可读。

## 2. 关键假设

- 本轮优先做前端解释层重组，尽量复用现有后端字段。
- 优先复用已有字段：
  - `why_selected`
  - `authority_tags`
  - `pending_authority_tags`
  - `authority_reasons`
  - `quality_flags`
  - `search_summary`
  - `search_diagnostics`
  - `source_statuses`
- 不追求一次性新增复杂打分体系。
- 先解决“用户看不懂结果为什么这样”的问题，不优先扩来源。

## 3. 核心目标

增强后，首页搜索结果应能直接解释四件事：

1. **为什么命中**
   - 这篇文献和本次搜索问题的关系是什么
   - 优先命中的原因是什么

2. **为什么推荐**
   - 是文献相关性高
   - 还是来源质量高
   - 还是方法/问题与当前搜索更贴合

3. **来源质量说明**
   - 哪些标签是已核验
   - 哪些只是待核验 / 推断
   - 为什么出现这些标签

4. **风险与异常提示**
   - 某来源空结果是“确实无结果”还是“限流/超时”
   - 当前这次搜索是否存在明显来源缺口

## 4. 分阶段实施

### 阶段 1：结果卡片解释信息收口

目标：

- 每篇文献卡片都更明确地解释“为什么命中 / 为什么推荐”。

重点：

- 统一 `why_selected` 的展示位置；
- 把 `authority_reasons` 从分散文本变成更容易扫读的解释区；
- 区分“相关性解释”和“来源质量解释”。

### 阶段 2：搜索结果总览解释增强

目标：

- 用户在结果页顶部就能看懂这次检索整体质量。

重点：

- 对 `search_summary` 做更明确的信息分层；
- 单独解释“已核验 / 待核验 / 来源异常”；
- 把 `source_statuses` 和 `search_diagnostics` 收成一个统一的来源状态面板。

### 阶段 3：标签语义标准化

目标：

- 避免“官方核验”和“系统推断”混在一起。

重点：

- 为权威标签定义用户可见语义；
- 明确：
  - 已核验标签
  - 待核验标签
  - 来源推断标签
- 文案避免让用户误解为“官方目录已认证”。

## 5. 建议修改文件

- `frontend/src/components/HomeSearchLanding.tsx`
- `frontend/src/components/TopicResearchPanel.tsx`

  - `backend/app/services/authority_filter_service.py`

## 6. 验收标准

- 用户能看懂单篇文献为什么出现在结果里；
- 用户能区分“高相关”和“高权威”不是一回事；
- 用户能明确哪些标签是已核验，哪些只是待核验；
- 用户能看懂来源异常是“空结果”还是“限流/超时/失败”；
- 首页结果页的信息结构更统一，不再分散。
