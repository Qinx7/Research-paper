# 用户认证（JWT）— 总纲

## 目标

实现基于 JWT 的用户注册/登录系统，并为现有 API 添加多用户隔离和访问控制。

## 当前状态

- `passlib[bcrypt]` 和 `python-jose[cryptography]` 依赖已安装但从未使用
- `JWT_SECRET_KEY` 等配置已定义但未被任何代码读取
- 所有 API 端点完全开放，无任何访问控制
- 所有数据模型无 `user_id` 字段，无用户归属概念

## 核心场景

1. 用户注册账户 → 登录获取 Token → 所有 API 请求携带 Token
2. 用户只能看到和管理自己的项目、文献、草稿等数据
3. 访客无法访问任何受保护的 API

## 架构概览

```
注册/登录请求
    ↓
POST /api/auth/register  →  User 表 写入
POST /api/auth/login     →  验证密码 → 返回 JWT Token
    ↓
后续所有 API 请求携带 Authorization: Bearer <token>
    ↓
Depends(get_current_user) 解析 Token → 注入当前 User
    ↓
路由中使用 current_user 进行权限过滤和数据隔离
```

## 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 认证方式 | JWT Bearer Token（对称 HS256） | 无状态，无需服务端 session |
| 保护方式 | FastAPI `Depends()` 依赖注入 | 比中间件灵活，支持按路由选择性保护 |
| 密码哈希 | bcrypt（passlib） | 依赖已安装，业界标准 |
| 用户标识 | email + username + password | 学术场景 email 为主 |
| 用户隔离 | Project 表添加 `user_id` FK | 以项目为隔离边界，子资源通过 project 间接归属 |
| 旧数据处理 | 现有 Project 数据 `user_id` 设为 NULL | 不破坏现有数据，后续可手动迁移 |
| Token 有效期 | 24 小时（1440 分钟） | 与现有配置一致 |

## 不在此范围

- 刷新 Token / Token 续期
- 邮箱验证
- 密码重置
- OAuth（ORCID / Google / GitHub）
- 角色权限（admin / student / advisor）
