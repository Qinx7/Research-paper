# 阶段1 01 风格方案与接口

## 目标

先把“风格”从固定实现里抽离出来，形成清晰的配置层和接口层，为后续页面渲染改造提供稳定输入。

## 最小实现原则

本步骤只做结构设计与接口梳理，不追求一次性把所有页面渲染逻辑全部重写。

## 计划内容

### 1. 梳理风格对象结构

建议在后端定义统一风格配置结构，至少包含：

- `id`
- `name`
- `description`
- `palette`
- `fonts`
- `layout_traits`

其中：

- `palette` 负责颜色体系；
- `fonts` 负责标题/正文的字体与字号；
- `layout_traits` 负责封面是否居中、章节页是否大色块、内容页是否卡片化等风格差异。

### 2. 定义首批风格预设

建议首批内置以下风格：

- `academic_blue`
- `minimal_gray`
- `tech_dark`
- `vibrant_orange_green`

### 3. 扩展 PPT 生成请求

现有请求中的 `template` 字段继续保留，但在实现中视为风格标识。

输入策略：

- 传入合法风格 ID：按该风格生成；
- 未传：使用默认风格；
- 传入未知值：记录日志并回退默认风格。

### 4. 增加风格列表接口

建议新增只读接口，例如：

- `GET /api/ppt/styles`

返回：

- 风格 ID
- 风格名称
- 简短说明
- 适用场景
- 是否默认

## 验收标准

- 后端具备统一风格定义入口；
- 至少有 4 套风格元数据；
- 前端可以通过接口或常量拿到可选风格列表；
- 旧调用仍可继续使用默认风格生成。
