# guizang-ppt-skill 适配方案总纲

## 1. 需求理解

目标不是把 `op7418/guizang-ppt-skill` 原样搬进当前项目，而是判断它如何接入我们已经建立的：

- `SkillDefinition`
- `SkillRegistry`
- `SkillExecutor`
- 现有 `orchestration / workflow`

体系，并明确：

1. 它能不能直接接；
2. 适合接到哪一层；
3. 会不会冲击当前 `.pptx` 主链路；
4. 许可证是否允许当前项目直接深度集成。

## 2. 已核对的事实

基于仓库公开说明，可以确认：

1. 这是一个面向 AI Agent 的 skill，而不是 Python SDK 或 Node SDK。
2. 主要产物是 **HTML slide deck / web PPT**，不是 `.pptx`。
3. 仓库强调：
   - 单文件 HTML
   - 浏览器直接打开
   - Agent 按 `SKILL.md + assets + references` 工作流执行
4. 仓库许可证是 **AGPL-3.0**。

## 3. 关键结论

### 3.1 不能直接作为我们现有 PPT skill 的同构替代

原因：

当前项目里的 PPT 主链路是：

- `ppt_agent.py`
- `defense_ppt_agent.py`
- `python-pptx`
- 下载 `.pptx`

而 `guizang-ppt-skill` 的核心链路是：

- `SKILL.md`
- HTML 模板
- 版式规则
- 浏览器展示

所以它不能直接替换现有：

- `ppt.proposal_render`
- `ppt.defense_render`

### 3.2 可以接入我们的技能系统，但应作为新 skill 类型

最合理的定位是：

- 新增一类 **HTML deck / Web PPT** 技能

而不是硬塞进当前 `.pptx` render skill。

建议 skill id：

- `ppt.web_html_deck`

## 4. 推荐接入策略

推荐采用：

**“并行接入，不替换主链路”**

即：

1. 保留现有 `.pptx` 交付链路；
2. 新增一个实验型或增强型 HTML deck skill；
3. 先服务：
   - 路演版演讲 deck
   - 个人展示版分享 deck
   - 答辩预演版 HTML deck

不一开始替换正式答辩材料下载链路。

## 5. 新 skill 的建议定位

### 5.1 新增 skill

建议至少新增：

- `ppt.web_html_deck`
- `ppt.web_cover_assets`

其中：

#### `ppt.web_html_deck`

职责：

- 根据内容与模板生成 HTML slide deck

#### `ppt.web_cover_assets`

职责：

- 生成公众号头图 / 分享卡 / 相关视觉封面

## 6. 与当前 skill 系统的映射

### 6.1 Skill Definition

对 `guizang-ppt-skill` 的适配，不是直接读取 `SKILL.md` 执行，而是把它包装成我们的 skill definition：

- `id`
- `name`
- `description`
- `domain`
- `input_schema`
- `output_schema`
- `handler`
- `guards`

### 6.2 Handler

handler 的职责不是“解释 SKILL.md”，而是：

1. 读取我们的输入内容；
2. 选择 HTML 模板；
3. 生成 HTML deck 文件；
4. 产出可预览和可下载对象。

### 6.3 Output Schema

不能继续沿用当前 `.pptx` 输出结构。

建议输出：

```json
{
  "artifact_type": "html_deck",
  "title": "答辩预演版 HTML Deck",
  "object_key": "generated/decks/xxx/index.html",
  "preview_url": "/api/generated/preview/...",
  "download_url": "/api/generated/download/...",
  "cover_assets": []
}
```

## 7. 目录结构建议

推荐新增：

```text
backend/app/skills/definitions/ppt_web_html_deck.py
backend/app/skills/definitions/ppt_web_cover_assets.py
backend/app/services/web_deck_render_service.py
backend/app/services/web_deck_storage_service.py
```

如果后续真的引入它的模板资源，再考虑：

```text
backend/app/resources/guizang/
├── template.html
├── template-swiss.html
├── references/
└── scripts/
```

## 8. 适配方式分级

### 方案 A：轻量“思路借鉴”适配（推荐）

内容：

- 借鉴其 HTML deck 思路
- 借鉴其页面版式与模板结构
- 不直接把整个仓库作为运行时依赖

优点：

- 法律风险低
- 技术可控
- 与当前技能系统更容易融合

缺点：

- 无法直接复用其全部生态

### 方案 B：外部模板资源包装适配

内容：

- 把它的模板资源作为外部素材引入
- 我们写自己的 wrapper handler

优点：

- 视觉结果更接近原项目

缺点：

- 许可证风险更高
- 需要仔细处理 AGPL 影响

### 方案 C：直接深度集成为主链路

不推荐。

原因：

- 会直接冲击现有 `.pptx` 主链路
- 与当前答辩 / 开题正式交付不一致
- 许可证与维护成本都偏高

## 9. 许可证风险

当前最需要明确的是：

- `guizang-ppt-skill` 仓库为 **AGPL-3.0**

因此如果你打算：

1. 直接复制其核心模板与结构代码；
2. 把其仓库作为运行时一部分分发；
3. 在产品内深度集成并作为正式功能交付；

就必须先做许可证评估。

所以当前推荐结论是：

- **第一阶段只做“借思路的轻适配方案”**
- **不建议立即做深度源代码级集成**

## 10. 第一阶段最小实现目标

推荐第一阶段只做：

1. 在 skill system 中新增 `ppt.web_html_deck` 定义；
2. handler 先输出简单 HTML deck；
3. 不接正式答辩下载链路；
4. 只作为实验型或增强型输出；
5. 前端先提供“预览 HTML deck”入口，不替换原 PPT 按钮。

## 11. 不做事项

第一阶段不做：

1. 不替换现有 `python-pptx` 主链路
2. 不直接复制整个外部 skill 仓库到正式运行时
3. 不把 HTML deck 当作唯一答辩交付形式
4. 不做用户侧 skill marketplace

## 12. 验收标准

第一阶段完成后，应达到：

1. 技术上可在技能系统中注册 `ppt.web_html_deck`
2. 能输出一个 HTML deck 产物
3. 不影响现有 `.pptx` 生成
4. 许可证风险已在方案中明确标注

## 13. 需要确认

建议先按 **方案 A：轻量“思路借鉴”适配** 推进，不直接引入原仓库作为运行时依赖。

如果确认继续，下一步建议执行：

`阶段1：HTML deck skill 最小适配骨架`
