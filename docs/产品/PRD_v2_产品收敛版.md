# 文献驱动科研助手 PRD v2

## 1. 文档目的

本版本用于替代原始“大而全”式 PRD 的实施角色，作为当前项目后续迭代的产品基线文档。

原始 PRD 保留为愿景文档；本版本聚焦已经被真实使用验证过的主链路、页面边界、优先级和技术约束。

## 2. 产品定位

### 2.1 一句话定位

这是一个面向研究生科研场景的 **文献检索、选题分析、项目知识沉淀与论文写作辅助产品**。

### 2.2 当前阶段真实主价值

当前产品最稳定、最清晰的价值不是“覆盖科研全过程的一切能力”，而是以下闭环：

1. 用户在首页输入研究问题并发起学术检索
2. 系统返回多来源文献结果与结构化检索总结
3. 系统把文献、项目成果、证据卡片、草稿章节逐步沉淀到项目知识底座
4. 用户在项目页和写作页中继续完成知识整理、选题分析和论文写作

### 2.3 不再作为当前阶段核心定位的内容

以下内容保留为中长期能力，不再作为当前版本 PRD 的核心承诺：

1. 导师/团队协作平台
2. 全文检索引擎平台
3. 通用多 Agent 编排可视化平台
4. 覆盖科研全过程的“万能工作台”

## 3. 目标用户

### 3.1 当前主用户

1. 准备选题或准备开题的研究生
2. 已经开始做项目、需要整理文献与成果的研究生
3. 正在写论文、需要依据追溯与章节生成辅助的研究生

### 3.2 次级用户

1. 指导教师
2. 课题组内部协同成员

说明：
次级用户当前不作为本版本核心交付对象，相关能力进入中长期规划。

## 4. 当前版本产品边界

### 4.1 首页

首页只承担以下职责：

1. 文献搜索入口
2. 本次搜索结果展示
3. 本次搜索的需求理解、文献分析、研究方向三步内容
4. 搜索记录恢复

首页不再承担：

1. 通用聊天页角色
2. 多模块导航聚合工作台角色
3. 项目管理主入口角色

### 4.2 项目页

项目页承担以下职责：

1. 项目概览
2. 项目知识工作台
3. 项目文献库
4. 文献矩阵
5. 论文工作流入口
6. 知识图谱入口
7. Zotero 导入入口

### 4.3 写作页

写作页承担以下职责：

1. 章节编辑
2. 当前章节知识依据展示
3. 章节生成
4. 合规检查
5. 导出与答辩衔接

### 4.4 研究页

研究页承担以下职责：

1. 研究方向结果承接
2. 项目设计结果承接
3. 后续开题材料主流程入口预留

## 5. 核心业务主链路

### 5.1 主链路 A：搜索到选题

```text
首页输入研究问题
↓
多来源学术检索
↓
返回文献结果与来源诊断
↓
自动生成需求理解
↓
自动生成文献分析
↓
自动生成研究方向候选
↓
用户保存研究方向到项目
```

### 5.2 主链路 B：项目沉淀到写作

```text
项目页查看文献、成果和知识状态
↓
成果文件解析入知识库
↓
文献笔记与证据卡片沉淀
↓
论文草稿创建
↓
章节生成与编辑
↓
依据追溯、合规检查、导出
```

### 5.3 主链路 C：写作到答辩

```text
草稿章节完成
↓
摘要生成
↓
答辩大纲生成
↓
答辩材料导出
```

## 6. P0 功能

### 6.1 学术搜索

必须满足：

1. 首页搜索稳定可用
2. 支持中英文范围切换
3. 支持多来源检索
4. 支持搜索结果缓存恢复
5. 支持来源状态诊断
6. 支持历史搜索删除

### 6.2 三步分析

必须满足：

1. 需求理解
2. 文献分析
3. 研究方向生成

要求：

1. 再次打开搜索记录时，不重复生成已有结果
2. 结果可恢复、可保存、可进入项目

### 6.3 项目知识底座

必须满足：

1. 成果上传
2. 成果解析入知识库
3. 项目知识概览可见
4. 文献笔记/证据卡片可见
5. 项目文献库可见

### 6.4 论文写作

必须满足：

1. 创建草稿
2. 章节生成
3. 章节编辑
4. 当前章节知识依据可见
5. 合规检查可运行
6. DOCX/PDF 导出可用

### 6.5 基础对象删除

必须满足：

1. 首页搜索记录可删除
2. 项目可删除
3. 删除项目时关联对象被正确清理

## 7. P1 功能

### 7.1 开题主流程产品化

目标：

1. 需求理解 → 文献分析 → 研究方向 → 项目设计 连续入口
2. 研究方向结果与项目设计结果形成连续承接
3. 开题报告与开题 PPT 接入口统一

### 7.2 学术搜索质量增强

目标：

1. 更多来源聚合
2. 来源相关性优化
3. 权威标签更可解释
4. 中文来源质量提升

### 7.3 写作依据动作闭环

目标：

1. 写作页能跳转到文献/成果/项目知识来源
2. 项目页能看见哪些知识被章节引用
3. 草稿引用和章节引用来源一致

## 8. P2 功能

### 8.1 答辩与交付工作台

1. 答辩 PPT 交付入口
2. 材料导出入口统一
3. 交付状态可见

### 8.2 异步任务体验

1. 任务状态更细
2. 失败重试与历史可见
3. 任务进度对用户更友好

## 9. 暂不纳入当前版本承诺

以下内容不再写进当前版本“核心功能”，避免产品目标发散：

1. 导师批注
2. 团队共享与权限分层
3. Elasticsearch/OpenSearch
4. LangGraph/LlamaIndex 重构
5. 全量数据库迁移体系
6. 通用开发者执行记录工作台

## 10. 数据源策略

### 10.1 当前已接入方向

1. OpenAlex
2. Semantic Scholar
3. Crossref
4. arXiv
5. PubMed
6. CNKI
7. 维普
8. PubScholar

### 10.2 数据源接入原则

1. 优先真实可用
2. 优先来源可解释
3. 接口不可直连时允许页面会话抓取
4. 不为了“来源数量”牺牲稳定性

### 10.3 来源限制必须显式说明

PRD 必须承认：

1. 不同来源存在限流、反爬和授权限制
2. 部分来源不能承诺稳定返回
3. 权威标签不等于官方目录认证

## 11. 技术现实约束

### 11.1 当前架构原则

1. 不新增重型基础设施优先
2. 先做最小闭环
3. 先做兼容旧库旧数据
4. 先做用户主链路，不先做内部观测页面

### 11.2 当前必须承认的现实

1. 还没有 Alembic 迁移体系
2. 需要轻量 schema compatibility 补丁保证旧库可运行
3. 前端存在多处页面历史演进，需要逐步收敛状态模型

## 12. 兼容与迁移策略

### 12.1 数据库兼容

要求：

1. 新字段加入后，旧库必须有兼容补丁
2. 不允许把“代码已写新字段”直接假设为“数据库已存在”

### 12.2 前端缓存兼容

要求：

1. 搜索历史结构升级时，旧缓存可迁移
2. 恢复逻辑优先命中本地快照
3. 不允许再次打开历史记录时无意义重算

## 13. 成功标准

当前版本不是看“功能数量”，而是看以下是否稳定：

1. 首页搜索一次后，可重复打开历史记录且不重跑分析
2. 项目知识工作台可见、可操作
3. 草稿写作状态一致，引用与依据可追溯
4. 用户删除项目不会被外键问题阻塞
5. 研究方向、项目设计等结果可持续保存，不因旧库结构报错

## 14. 后续建议

后续所有新功能评审，先问三个问题：

1. 它是否直接增强主链路？
2. 它是否建立在稳定状态模型上？
3. 它是否会引入新的旧库/旧缓存兼容风险？

如果三个问题不能同时回答清楚，就不应进入当前版本开发。
